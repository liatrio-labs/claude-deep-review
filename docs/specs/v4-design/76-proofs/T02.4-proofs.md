## Implementation Summary

### validate_diff_lines() (R02.4.1, R02.4.2)

Replaced the stub (single-line check) with a full range check:

1. For each line in `[line_start, line_end]`, checks if `(filepath, line)` is in the `valid_lines` set from `parse_diff_lines()`
2. If any line in the range is in the diff: `diff_validation.in_diff = True`, finding kept with current origin
3. If NO lines in range are in the diff: `diff_validation.in_diff = False`, finding tagged as `origin = "surfaced"` (NOT eliminated)
4. Severity downgrade applied on diff-surfaced findings (same as blame-surfaced)
5. Always returns `True` — diff validation no longer eliminates findings

This fixes the calcom-PR10600 pattern where F-01 targeted line 121 (outside diff) and was wrongly eliminated. Now it survives as "surfaced".

### batch_findings() (R02.4.3)

Replaced the stub (sequential batches of 10) with file-proximity grouping:

1. Sort findings by `(file, line_start)` for stable file proximity ordering
2. Greedily fill batches: break when `max_batch=5` reached OR file changes AND current batch >= `min_batch=3`
3. Merge tiny tail (< min_batch) into previous batch if combined size <= max_batch
4. Returns list of ID-lists (not finding objects) — orchestrator references by ID

Output format: `"batches": [["id-1", "id-2", "id-3"], ["id-4", "id-5"], ...]`

### Main pipeline update

Phase 4 loop simplified: since `validate_diff_lines()` always returns `True`, the elimination branch was removed. A counter tracks how many findings were newly tagged as surfaced by diff validation.
