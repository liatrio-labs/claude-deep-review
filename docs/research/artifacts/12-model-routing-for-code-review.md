# Model routing in AI code review: more intuition than evidence

**The field of multi-agent code review is converging on task-based model routing—assigning different LLM tiers to different review subtasks—but the empirical foundation for these routing decisions is thinner than vendor marketing suggests.** Production systems like CodeRabbit, Ellipsis, and Anthropic's code review tools all use multi-model architectures, yet their routing logic is largely driven by internal benchmarking on proprietary datasets rather than published, peer-reviewed evidence specific to code review. The most rigorous academic work on model routing (RouteLLM, FrugalGPT) demonstrates **2–5× cost savings** while preserving quality on general benchmarks, but none evaluates code review subtasks directly. Google DeepMind's scaling study provides the strongest evidence that heterogeneous agent teams genuinely outperform homogeneous ones—but only in specific architectures. What follows maps the empirical terrain, separates validated findings from conventional wisdom, and proposes a practical routing framework grounded in what the evidence actually supports.

## What production systems actually do with model routing

The premise that Anthropic's code review plugin "uses Opus for bug detection and Sonnet for compliance" turns out to be **incorrect**. The plugin's architecture deploys **five parallel Sonnet agents** handling CLAUDE.md compliance, bug detection, git history, PR comment review, and code comment verification—all at the same model tier. The only Opus usage is in Anthropic's *separate* security review GitHub Action, which defaults to Claude Opus 4.1. This distinction matters: even Anthropic, with direct access to its own model internals, chose a uniform Sonnet tier for general code review and reserved Opus only for dedicated security analysis. No published benchmarks from Anthropic compare model tiers on code review subtasks.

**CodeRabbit represents the most transparent multi-model routing architecture in production.** Their pipeline uses NVIDIA Nemotron 3 for context gathering and summarization, OpenAI's **o3/o4-mini** for reasoning-heavy deep review (multi-line bugs, cross-file architecture issues), and **GPT-4.1** for routine QA checks and review summarization. Their routing is semi-dynamic: pipeline stages are architecturally fixed, but model assignments within stages are continuously re-evaluated using a standardized harness of 25 hard PRs with known error patterns. CodeRabbit's published benchmarks provide some of the only code-review-specific model comparisons available: GPT-5 found **254 of 300 known bugs (85%)**, outperforming Claude Opus 4 at roughly 200–207. Opus 4.5 earned the highest per-comment precision with ~25% fewer output tokens—described as "the architect reviewing your PR," producing surgical, decisive fixes while Sonnet 4.5 "hedges in 34% of actionable comments." Notably, CodeRabbit explicitly argues against exposing model selection to users: "Every product that puts a 'Choose your LLM' dropdown front and center is outsourcing an engineering decision to the least equipped person to make it."

Ellipsis AI runs **dozens of parallel "Comment Generator" agents**, mixing Claude Sonnet 3.6 (primary) with GPT-4o (parallel and fallback) and o1 (selectively for harder reasoning). Their architecture is both task-type-based and redundancy-based—different generators can use different models, and running multiple models in parallel provides coverage through diversity. A multi-stage filtering pipeline then deduplicates, confidence-filters, and hallucination-checks the raw outputs. HAMY's 9-agent system takes the opposite approach: **all agents run the same Claude model**, relying entirely on prompt specialization rather than model differentiation. Qodo 2.0 uses GPT-5 and Gemini 2.5 Pro as defaults with fallback chains via LiteLLM, representing a hybrid static-plus-fallback pattern.

The dominant pattern across production systems is **static task-based assignment**: reasoning-heavy tasks get frontier models, summarization gets mid-tier or specialized models, and routing decisions are made by engineering teams based on internal benchmarks—not by dynamic classifiers at runtime. True dynamic per-query routing exists mainly at platform level (GitHub Copilot's opaque internal routing, Amazon Bedrock's Intelligent Prompt Routing) rather than in individual code review tools.

## Academic evidence on routing effectiveness is promising but not code-specific

**RouteLLM**, published at ICLR 2025, trains lightweight routers on human preference data from Chatbot Arena to classify each query as needing a "strong" or "weak" model. It achieves **85% cost reduction on MT Bench** while maintaining 95% of GPT-4's performance, using only 26% GPT-4 calls via a matrix factorization router. However, RouteLLM's evaluations cover general benchmarks (MT Bench, MMLU, GSM8K)—no code-specific routing results are broken out. **FrugalGPT** (Stanford) takes a cascade approach, sequentially querying models from cheapest to most expensive until a DistilBERT scoring function deems the answer sufficient, achieving **up to 98% cost reduction** on some datasets. A key insight from FrugalGPT: "Even an inexpensive LLM can sometimes correctly answer queries on which a more expensive LLM fails"—model diversity has intrinsic value beyond cost savings.

The **DeepMind/Google scaling study** ("Towards a Science of Scaling Agent Systems," December 2025) tested 180 agent configurations across five architectures, three model families, and four benchmarks. Its findings on heterogeneous agents are nuanced and architecture-dependent. In **decentralized architectures** (peer debate with voting), mixed-capability teams consistently matched or outperformed homogeneous high-capability baselines across all three model families—OpenAI heterogeneous scored 0.53 vs. homogeneous 0.50; Anthropic scored 0.47 vs. 0.37. In **centralized architectures** (orchestrator-subagent), heterogeneity benefits were model-family-dependent: Anthropic models uniquely benefited, with a low-capability orchestrator plus high-capability subagents outperforming the homogeneous high-capability baseline by **31%**. The practical implication: "If you're spending money selectively, spend it on the workers, not the manager." The Together AI **Mixture-of-Agents** paper reinforces this, showing that heterogeneous models contribute significantly more than clones of the same model—"models generate better responses when they have access to outputs from other models, even if those outputs are of lower quality."

For **parallelizable tasks** (like code review, where multiple files can be reviewed independently), the DeepMind study found multi-agent systems improved performance by **+81%** over single agents. But for sequential tasks, multi-agent variants degraded performance by 39–70%. This validates the parallel agent architecture that most code review tools already use.

## The SWE-bench evidence on model tiers is both clarifying and humbling

SWE-bench Verified scores as of early 2026 show **remarkable convergence at the frontier**: Claude Opus 4.5 (80.9%), Gemini 3.1 Pro (80.6%), MiniMax M2.5 (80.2%), GPT-5.2 (80.0%), and Claude Sonnet 4.6 (79.6%) cluster within **1.3 percentage points** of each other. The gap between Opus and Sonnet—once the primary justification for routing critical tasks to frontier models—has compressed to just 1.2 points on this benchmark. Claude Haiku 4.5 scores 73.3%, which "would have been state-of-the-art just six months ago." This compression undermines the intuition that frontier models are categorically necessary for complex code tasks.

**SWE-bench Pro** (harder, enterprise-grade tasks) tells a dramatically different story. Here, GPT-5.3-Codex leads at 56.8%, but GPT-4o collapses to **4.9%**—a chasm that validates the conventional wisdom that task difficulty determines whether the frontier premium is justified. The agent scaffold matters enormously: a mid-tier model in a well-designed harness routinely beats a frontier model in a poor one, with **22+ point swings** attributable to scaffolding alone on SWE-bench Pro.

For **security vulnerability detection** specifically, the evidence is sobering. A UPenn study across 16 models found mean accuracy of **~62.8%** on vulnerability detection. No single model dominates across all vulnerability types. Perhaps most striking: smaller models sometimes outperform larger ones (Qwen-2.5-14B and Qwen-2.5-32B reported higher accuracy than GPT-4 on real-world datasets). Semgrep's real-world evaluation found **true positive rates of just 14% for Claude Code and 18% for OpenAI Codex** across real codebases—with false positive rates above 80%. The SecLLMHolmes study concluded bluntly: all current LLMs are "unreliable" for autonomous vulnerability detection.

## Which subtasks actually need which model tier

The conventional wisdom—"creative reasoning needs Opus, structured evaluation needs Sonnet, pattern matching needs Haiku"—is a useful heuristic but an oversimplification. The evidence better supports framing by **task complexity and ambiguity** rather than a creative/analytical split.

**Frontier-tier (Opus-class) advantages are empirically supported for:**

- Complex multi-file refactoring over 10K+ changed lines, where cross-reference tracking degrades in smaller models
- Security audits requiring deep inter-procedural data-flow analysis (Opus 4.1 achieves ~50% "important" comment share in CodeRabbit's benchmarks vs. Sonnet 4.5's 41.5%)
- Ambiguous or underspecified problems where the model must make defensible assumptions
- Long-horizon autonomous tasks where Opus achieves higher pass rates with up to 65% fewer tokens
- SWE-bench Pro-level tasks where the performance gap between tiers widens dramatically

**Mid-tier (Sonnet-class) is demonstrably sufficient for:**

- Standard bug detection (Sonnet 4.6 at 79.6% on SWE-bench Verified, 1.2 points behind Opus)
- Test generation, where "the quality delta compared to Opus is negligible"
- Code review triage—87% of Sonnet 4.5's actionable comments included code blocks or patches
- Documentation review and CI/CD scripting
- Most routine feature implementation, where output is "often indistinguishable from Opus for medium-complexity tasks"

**Fast-tier (Haiku-class) handles adequately:**

- Linting and style checking (explicitly recommended by Anthropic's own documentation)
- Simple syntax validation and code formatting
- High-frequency sub-agent tasks at 90% of Sonnet's agentic capability for one-third the cost
- Initial triage to classify change complexity before routing to more expensive models

The critical caveat: **no formal benchmark exists comparing model tiers specifically on code review subtasks** like "convention checking vs. security review vs. test coverage analysis." The tier recommendations above are synthesized from SWE-bench data, CodeRabbit's proprietary benchmarks, and practitioner reports—not from controlled experiments isolating code review subtask types.

## Model-specific blind spots and calibration create real routing implications

Different models exhibit **dramatically different vulnerability-class detection profiles**, which is perhaps the strongest empirical argument for multi-model routing. Semgrep's real-world testing revealed that Claude Code found IDOR bugs at 22% true positive rate but SQL injection at only 5%, while OpenAI Codex found **0% IDOR and 0% SQLi but 47% for path traversal**. SecVulEval found that CWE-362 (race conditions) was detected only by Claude-3.7-Sonnet and GPT-4.1; CWE-20 (improper input validation) was detected only by GPT-4.1; and CWE-120 (buffer overflow) was detected by **no model tested**. This vulnerability-class complementarity is the most empirically grounded case for running multiple models on security-sensitive code.

All frontier models are **systematically overconfident** when verbalizing confidence scores. An ICLR 2024 study found LLMs express confidence predominantly in the 80–100% range in multiples of 5, mimicking human confidence expression patterns. Claude Opus 4.5 showed the best calibration among frontier models (ECE=0.120 on KalshiBench), but still with substantial error. RLHF amplifies overconfidence because reward models have inherent biases toward high-confidence outputs regardless of quality. Reasoning models (extended thinking) may actually worsen calibration—"longer reasoning chains may reinforce initial hypotheses rather than genuinely updating on evidence." For confidence scoring in code review, this means **raw model-reported confidence should not be trusted** without external calibration.

On multi-model agreement: the **EnsLLM study** found ensembles of 14 LLMs achieved 90.2% accuracy on HumanEval vs. 83.5% for the best single model. When at least two models independently generated correct solutions, the ensemble always selected the correct one. The **SE-Jury** approach matched human inter-rater reliability on code generation and repair tasks. However, a critical March 2025 paper warns that "models agree with each other more reliably than they agree with truth"—model errors are correlated across families, partially violating the independence assumption. The paper concludes that aggregation improves performance **when a verifier exists** (like test execution) but "cannot serve as a verifier substitute." For code review, this means multi-model agreement strengthens bug-finding signals when tests can confirm the finding, but provides weaker guarantees for subjective assessments like code quality.

## Practical routing architectures and their tradeoffs

Four routing architectures exist in practice, with clear tradeoffs:

- **Static routing** (hardcoded model per agent): Used by most code review tools. Simple, predictable, easy to debug. CodeRabbit, Anthropic's plugin, and HAMY all use variants. The downside is no adaptation to per-query complexity—every style check pays frontier-model prices if that's the assigned tier.
- **Dynamic routing** (classifier selects model per query): RouteLLM and Martian demonstrate this. Achieves 2–5× cost savings on general benchmarks. Amazon Bedrock offers built-in routing within model families with up to 30% savings. Adds 100–300ms latency per request for the routing decision.
- **Cascade routing** (sequential escalation): Start with the cheapest model; if a quality check fails, escalate. FrugalGPT pioneered this. ETH Zurich's unified approach combining routing and cascading improves cost-performance by up to **14%** over either alone. Higher latency for complex queries but optimal cost for easy ones.
- **Redundant routing** (run multiple models, take best): Ellipsis's parallel generators exemplify this. Highest quality ceiling but highest cost. The Mixture-of-Agents approach shows synthesis of multiple outputs outperforms selection of the best single output—fusion beats ranking.

Cost implications are significant. At current pricing, Claude Opus 4.6 costs $5/$25 per million input/output tokens; Sonnet costs $3/$15; Haiku costs $1/$5; and budget models like DeepSeek V3 cost $0.28/$0.42. A typical file review using 5–10K input tokens costs ~$0.05–0.08 at Opus rates and ~$0.005–0.015 at Haiku rates—a **5–10× difference per review**. Routing 90–95% of reviews through Sonnet with 5–10% escalated to Opus represents the practitioner-consensus cost optimization, though the specific ratio is anecdotal rather than empirically derived.

Context window size creates routing constraints that are often underappreciated. All models suffer from **context rot**—a Chroma study across 18 models found every model showed quality degradation with growing context, with middle-context accuracy dropping from 85–95% to 76–82%. Effective context windows are much smaller than advertised, with sharp performance drops past 32K tokens for most models. For code review, this means smaller models with limited windows need focused context engineering (only relevant files and functions), while models with larger effective windows (Gemini at 1M+) can process broader repository context. Research shows **context quality outperforms context quantity**: well-selected information consistently outperforms large, noisy inputs.

## A practical framework for model routing decisions

Based on the evidence gathered, here is a decision framework for model routing in a multi-agent code review system. Items marked with ✅ are empirically supported; items marked with ⚠️ are conventional wisdom with partial evidence.

**Tier 1 — Fast models (Haiku-class, $1–5/Mtok):**
Use for initial triage and complexity classification (⚠️), linting and style enforcement (⚠️, recommended by Anthropic but not formally benchmarked), PR summarization and changelog generation (✅, validated by CodeRabbit's Nemotron usage), and high-frequency sub-agent tasks where volume matters more than depth (✅, Haiku achieves 90% of Sonnet's agentic capability).

**Tier 2 — Mid-tier models (Sonnet-class, $3–15/Mtok):**
Use as the default for all code review agents (✅, Anthropic's own plugin uses all-Sonnet), standard bug detection and logic error identification (✅, within 1.2 points of Opus on SWE-bench Verified), test coverage analysis and test generation (✅, quality delta vs. Opus negligible), documentation review and code comment evaluation (⚠️), and as the orchestrator in centralized architectures (✅, DeepMind study shows orchestrator capability matters less than worker capability).

**Tier 3 — Frontier models (Opus-class, $5–25/Mtok):**
Reserve for security vulnerability analysis (✅, Anthropic defaults security review to Opus; CodeRabbit data shows higher important-share), complex cross-file architectural review on large PRs (⚠️, supported by SWE-bench Pro gap data), ambiguous or novel code patterns requiring deep reasoning (⚠️, partially supported by CodeRabbit's "hedging" data), and escalation targets in cascade routing when mid-tier confidence is low (✅, supported by FrugalGPT and RouteLLM).

**Multi-model diversity for security:** Run at least two models from different families on security-critical code (✅, strong empirical support from Semgrep's complementary vulnerability-class detection profiles). Do not trust single-model confidence scores—use external verification (tests, static analysis) to validate findings (✅, "consensus is not verification" paper).

**Architecture recommendation:** A cascade or hybrid architecture combining static task-based routing with complexity-triggered escalation captures most of the cost savings while preserving quality ceilings. Start every review with a Sonnet-class agent; route to Opus only when the task involves security analysis, cross-file complexity above a threshold, or when mid-tier confidence scoring falls below a calibrated cutoff. Use Haiku-class models for triage, summarization, and style checking. Combine LLM analysis with deterministic static analysis tools—hybrid approaches reduce false positives by **94–98%** in industrial settings.

## Conclusion

The honest summary is that **model routing for code review has far more practitioner intuition than rigorous, published evidence.** The strongest empirical finding is that different models have genuinely different vulnerability-class detection profiles, making multi-model diversity valuable for security review. The DeepMind scaling study validates heterogeneous agent teams for parallelizable tasks. SWE-bench data shows the frontier premium is small on standard tasks but large on hard ones—supporting a cascade architecture. But no controlled study compares model tiers specifically across code review subtask types (style vs. bugs vs. security vs. test coverage). CodeRabbit's proprietary benchmarks come closest but aren't independently reproducible. The practical implication: build routing architectures that are easy to re-evaluate as models evolve, because today's optimal routing table will be wrong within six months. The scaffold matters as much as the model—invest in context engineering, confidence calibration, and multi-stage filtering before optimizing which model serves which agent.
