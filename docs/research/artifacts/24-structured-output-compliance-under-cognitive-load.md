# Why AI agents abandon structured output under load — and how to fix it

**Format compliance and complex reasoning are empirically proven competing objectives in LLMs.** The failure you observe — Claude Code subagents drifting from NDJSON to natural language during complex investigations — is not a prompting bug but a fundamental architectural property of transformer attention. Research across IFScale, LongGenBench, and FollowBench benchmarks demonstrates that instruction adherence degrades predictably with task complexity and output length, with **clear decline beyond 4,000 tokens** and measurable U-shaped attention curves that starve mid-context format instructions. Production frameworks like Aider, Cursor, and CodeRabbit have independently converged on the same solution: separate investigation from structured emission entirely. Within Claude Code's subagent constraints, three implementable architectural patterns can resolve this, ordered by effectiveness.

## The empirical case: format compliance degrades measurably with complexity

The research literature validates this failure mode with striking specificity. **IFScale** (Jaroslawicz et al., 2025) scaled instruction density from 10 to 500 constraints and found that even frontier models achieve only **68% accuracy at high density**. Claude Sonnet 4 exhibits "linear decay" — a steady, predictable decline in instruction following as constraint count rises. More critically, the study found a **strong primacy bias**: earlier instructions are followed more reliably than later ones, meaning the NDJSON format instruction competes poorly against the investigation task described alongside it.

**LongGenBench** (ICLR 2025) directly measured instruction adherence during long-form generation at 16K and 32K tokens — matching the Claude Code subagent's 32K limit. The finding is unambiguous: instruction adherence and coherence decline beyond the **4,000-token threshold**, with further deterioration as outputs approach 16K tokens. Periodic instructions (like "emit JSON after each file") degrade faster than single-step instructions. About **45% of long outputs** exhibited significant repetition patterns, and adjusting generation parameters showed "limited success" in mitigation.

The mechanism is architectural. The "lost in the middle" effect (Liu et al., 2024, TACL) shows a U-shaped attention curve where models attend most to the beginning and end of sequences, with middle positions receiving diminished attention. As the subagent generates more investigation text, the system prompt's format instructions — positioned at the beginning — compete against an ever-growing body of generated content for attention resources. This is not a matter of "forgetting" in the human sense but a measurable property of transformer self-attention: layer-wise positional importance profiles show consistent peaks at primacy and recency positions, a pattern that persists even when input text is lexically scrambled.

The competing-objectives dimension adds another layer. The **Deco-G** paper (2025) formally demonstrated that "the entanglement [of format and reasoning instructions] creates competing goals for the model," with stricter format constraints causing **1–6% relative degradation in reasoning performance**. "Let Me Speak Freely?" (Tam et al., 2024, EMNLP) found **10–15% reasoning degradation** when LLMs were constrained to JSON output versus free-form generation. The implication is reciprocal: when reasoning gets harder, format compliance suffers — and this is exactly the pattern observed in complex 20+ file investigations.

## Production frameworks solved this by separating investigation from structure

Every major production agent framework that handles structured output from investigation agents has independently converged on variants of the same insight: **do not ask the model to investigate and emit structured data in the same generation pass.** The specific implementations vary, but the pattern is universal.

**Aider's Architect/Editor split** is the clearest example. When OpenAI's o1 models proved excellent at reasoning but terrible at edit formatting, Aider's solution was to create two distinct roles: an "Architect" LLM proposes solutions in free-form natural language, and a separate "Editor" LLM converts those proposals into structured file edits. This pairing of o1-as-Architect with Sonnet-as-Editor achieved state-of-the-art benchmark scores. Critically, Aider's benchmarks showed that **function calling and JSON format actually degraded code quality** compared to simple text formats — the JSON escaping overhead consumed model capacity that should have gone toward correct code generation.

**Cursor** reached the same conclusion from a different direction. Their engineering team found that "LLMs are terrible at generating diffs" — deterministic matching failed **40%+ of the time**. Their solution was full-file rewrites processed through a specialized "Fast Apply" model (fine-tuned Llama-3-70b) that applies edits at ~1,000 tokens/second. Their Bugbot code review system evolved from a pipeline architecture (8 parallel passes with majority voting, **52% resolution rate**) to a single agentic architecture with tool use (**70%+ resolution rate**), but the structured output still comes through tool calls rather than inline text.

**CodeRabbit** explicitly states in their engineering blog that "LLMs are particularly bad at understanding and generating structured content." Their architecture uses a multi-stage pipeline where simpler models triage changes, complex models perform deep review, and **post-generation verification scripts** validate and reformat the output. The structuring happens in post-processing, not by asking the LLM to emit structured data inline.

**PR-Agent/Qodo** takes a different approach — inline YAML in the LLM response — but compensates with extensive fallback parsing. Their `try_fix_yaml()` function applies multiple repair strategies: extracting YAML from code fences, adding block scalar indicators, handling diff-style symbols, and stripping problematic characters. When all repairs fail, the system retries with fallback models. The persistent YAML parsing issues across their GitHub issue tracker (#461, #1524, #1791, #1901) testify to the fragility of inline structured output even with YAML's more LLM-friendly syntax.

**OpenHands/OpenDevin** represents the tool-call-emission approach at its purest: all agent-environment interactions flow as typed events through an event-stream architecture, with the LLM generating native function/tool calls that are deserialized into typed Action classes. The model never writes inline structured text — structure is enforced by the tool-call schema itself.

## Three implementable patterns for Claude Code subagents

Given Claude Code's constraints — agents defined by `.md` files with YAML frontmatter, restricted tool sets (Read, Write, Edit, Bash, Glob, Grep), no custom tool definitions — three architectural patterns can address the compliance problem, each with different tradeoff profiles.

**Pattern 1: Parent-orchestrated micro-batching** decomposes the investigation into per-file or per-module subtasks, dispatching separate subagent invocations for each. Each invocation analyzes 1–3 files maximum with mandatory NDJSON output as the final step. The parent agent collects and concatenates all outputs. This mechanically prevents drift by keeping each output short — well under the 4,000-token threshold where compliance begins degrading. The cost is higher latency, more API calls, and potential loss of cross-file context. For the cross-file context problem, the parent can provide a summary of prior findings in each subagent's prompt. This pattern draws directly from the BAVT research finding that "without structured resource management, agents frequently exhaust token limits on redundant or low-yield steps."

**Pattern 2: Write-tool-as-emission-channel** separates the investigation stream (text output) from the structured emission stream (file writes). Instead of emitting JSON inline in its response, the subagent uses the `Write` tool to append NDJSON findings to a `findings.ndjson` file, or uses `Bash` with `echo '{"severity":"high",...}' >> findings.ndjson`. This is the closest available analogue to OpenHands' tool-call emission pattern. Research on external memory systems shows **21% higher instruction retention** when agents use structured logs versus relying solely on conversation history — the file write acts as a "behavioral anchor" resistant to attention drift. A `PostToolUse` hook on the `Read` tool can validate that the agent has written a corresponding finding after each file read, creating a mechanical feedback loop.

**Pattern 3: Two-phase dual subagent** follows Aider's Architect/Editor pattern. Subagent 1 (the "investigator") has Read, Grep, Glob, and Bash tools and investigates freely, writing natural-language notes to a scratch file. Subagent 2 (the "formatter") reads the investigation notes and produces NDJSON findings. Each agent has a focused, non-competing objective. The formatter's task is simple enough that format compliance should be high. The tradeoff is added latency and the risk that the formatter loses nuance from the investigation. The investigation notes should include explicit markers like file paths, line numbers, and severity assessments to minimize this loss.

## Prompting techniques that shift the compliance equilibrium

When architectural separation isn't feasible, prompting techniques can meaningfully shift the compliance probability — though research consistently shows no single technique is sufficient. The "Drift No More?" paper (2025) formalized format drift as a stochastic process with a non-zero equilibrium, but crucially showed that **targeted reminders shift that equilibrium**, reducing divergence measurably. The following techniques, used in combination, provide the strongest defense.

**Few-shot examples of the interleaved format** are likely the single highest-impact prompting technique for this use case. The interleaved investigation-then-JSON pattern is unusual — neither pure structured output nor pure prose — so the model needs to "see" what it looks like. Include 1–2 concrete examples in the `.md` file showing the exact rhythm: investigation prose analyzing a specific file, followed immediately by a JSON finding line, followed by more investigation. Research consistently shows few-shot prompting for format compliance is robust, with diminishing returns after 2–3 examples.

**XML tag structuring** exploits Claude's specific fine-tuning to attend to XML tags. Anthropic's documentation states Claude models "pay special attention to the structure created by XML tags." Defining output zones with `<investigation>` and `<finding>` tags creates structural anchors that resist attention dilution. A practitioner testing XML tags with Claude Code specifically found they "effectively isolate key elements of a prompt, preventing Claude Code from forgetting or ignoring the context," with complex prompts benefiting significantly more than simple ones.

**The sandwich technique** — placing format instructions at the beginning, middle, and end of the `.md` file — directly exploits the U-shaped attention curve. The recommended `.md` structure: role definition and format requirements at the top (primacy), detailed task instructions in the middle, 1–2 few-shot examples near the bottom, and a concise format reminder as the final line (recency). Lilian Li's research on format compliance specifically recommends repeating format requirements three times throughout prompts.

**Explicit state machine protocol** prompting defines a mechanical alternation between INVESTIGATE and EMIT states with explicit transition rules. The StateFlow paper demonstrated that formalizing LLM inference as a state machine improves task-solving over unstructured approaches. While LLMs don't literally compile state machines, the explicit structure "strongly biases execution" toward compliance. Define the protocol with numbered steps: (1) Read and analyze code, (2) IMMEDIATELY write findings as NDJSON, (3) Only then proceed to the next file. Include the rule: "Never analyze more than one file without emitting findings."

**Embedded reasoning fields** in the JSON schema allow the model to satisfy both the investigation and emission objectives simultaneously. Including a `reasoning` field as the first property in each JSON finding lets the model "think" within the structure. Empirical testing showed that placing a `reasoning_steps` field first in structured output schemas yielded "100% accurate answers nearly every time" versus 0% without it — the model needs to reason before producing the answer fields.

## The output budget is a red herring — but budget-aware techniques help

The 32K token limit is not the root cause. Research from the TALE paper (ACL 2025) found that an "ideal budget range" exists for each task — beyond it, additional budget yields diminishing returns and can even increase costs without improving quality. Moving from 32K to 64K may provide more runway but won't eliminate the fundamental competition between content generation and format compliance. The model isn't deliberately "saving" budget; rather, format compliance and investigation content compete for the same probabilistic generation resources, and this competition intensifies predictably with output length.

That said, budget-aware prompting can help. The "Drift No More?" paper showed that mid-generation reminders shift the drift equilibrium measurably. For Claude Code subagents, this translates to explicit budget reservation instructions: "Reserve the last 4,000 tokens of your output for structured JSON findings. Before reaching 80% of your output budget, begin emitting structured output." While this operates at the prompt level rather than mechanically, it provides an additional signal that counteracts the attention decay gradient.

**Extended thinking** offers a more architectural solution for the budget problem. Anthropic's extended thinking separates the thinking budget from the output budget — investigation can happen in thinking blocks (invisible to the parser) while structured output happens in the main response. The "think" tool, recommended by Anthropic specifically for agentic workflows, creates "dedicated space for structured thinking during complex tasks" and showed "remarkable improvements in Claude's agentic tool use ability." For Claude Code subagents, if extended thinking is available, instructing the agent to perform investigation in thinking and emit only NDJSON in the visible response could dramatically improve compliance.

## Conclusion

The structured output compliance failure in Claude Code subagents is not a solvable prompting puzzle but a measurable architectural property: **format constraints and complex reasoning compete for finite attention resources**, and this competition intensifies with output length and task complexity. The most robust solutions are architectural, not linguistic.

The highest-confidence recommendation is **micro-batching** (Pattern 1): decompose investigations into per-file subagent invocations that never generate enough output for drift to occur. This trades latency for guaranteed compliance. When latency is unacceptable, the **Write-tool-as-emission-channel** (Pattern 2) provides the best single-subagent approach by externalizing structured output to file writes — leveraging the 21% instruction retention advantage of external memory while using `PostToolUse` hooks as mechanical compliance validators.

For prompting within any pattern, combine four techniques simultaneously: few-shot examples of the exact interleaved format, XML tag structuring for output zones, sandwich-technique instruction repetition, and explicit state machine protocol definition. No individual technique is sufficient, but their combination shifts the drift equilibrium measurably toward compliance. The key architectural insight from production frameworks is unanimous: the industry has converged on separating investigation from structuring, not on making inline structured emission more reliable.
