# Optimal agent count for AI code review peaks near five

**Multi-agent code review hits diminishing returns fast: quality plateaus around 4–6 specialized agents, and unstructured systems amplify errors up to 17.2×.** The most effective architectures use a small set of coarse-grained concern specialists — security, correctness, performance, architecture, testing — coordinated by a centralized orchestrator with a dedicated aggregation layer. Google DeepMind's landmark December 2025 study across 14,742 runs established that coordination overhead consumes agent benefits beyond a saturation threshold, while production systems from Anthropic, Qodo, and others converge on hierarchical two-level designs with 5–9 domain agents. The research is clear: more agents is not better. The right agents, structured correctly, is what matters.

## Scaling laws follow a logistic curve with a hard ceiling

The most rigorous empirical evidence comes from Google DeepMind's "Towards a Science of Scaling Agent Systems" (Kim et al., December 2025, arXiv:2512.08296), which tested **180 configurations** across five agent architectures, three LLM families, and four benchmarks totaling 14,742 instance runs. The findings reshape how we should think about multi-agent system design.

Performance follows a **logistic (S-curve) growth pattern** as agent count increases. Initial agents provide substantial gains, but the curve flattens rapidly. The MacNet study (ICLR 2025), testing up to 1,000+ agents in mesh networks, confirmed this shape — performance grows slowly at first, accelerates, then hits a hard saturation point. For code review tasks, which are largely parallelizable, the inflection point arrives early.

DeepMind identified a critical **saturation threshold at approximately 4 agents**. Below that number, adding agents improves quality. Above it, coordination overhead consumes the benefits. They also discovered the "45% rule": once single-agent baseline accuracy exceeds roughly 45%, adding more agents yields diminishing or negative returns (β = −0.408, p < 0.001). This is termed "capability saturation" — when the base model is already competent, the coordination tax of multi-agent systems outweighs any ensemble benefit.

The error amplification data is striking. **Independent (unstructured) agent networks amplified errors 17.2× versus single-agent baselines.** Agents working in parallel without communication let errors cascade unchecked. Centralized orchestration — a hub-and-spoke supervisor pattern — contained amplification to **4.4×**, a fourfold improvement but still substantial. Decentralized peer-to-peer topologies fell between these extremes, with errors spreading rapidly through direct communication channels.

Token efficiency degrades sharply with agent count. A single agent completed roughly **67 successful tasks per 1,000 tokens**; a centralized four-agent system managed only 21; a hybrid architecture, just 14. The documented **3.5× token cost multiplier** for a four-agent system makes the economics clear — each additional agent must justify itself not just in quality but in cost per finding. Compound reliability math compounds the problem: at 95% per-step reliability across 10 sequential steps, system reliability drops to 59.9%.

The information-theoretic perspective from "Understanding Agent Scaling via Diversity" (February 2026, arXiv:2602.03794) explains why homogeneous agents saturate early. Performance is bounded by **intrinsic task uncertainty**, not agent count. Homogeneous agents — same model, same prompts — produce strongly correlated outputs, so adding more barely moves the needle. Heterogeneous agents (different models, prompts, or tools) delay saturation by contributing complementary, non-redundant information channels. The paper introduces K*, an "effective channel count" measuring non-redundant information sources, which is the real determinant of multi-agent value.

## Structured topologies and centralized orchestration prevent error cascades

Three architectural patterns demonstrably prevent the 17.2× error amplification of unstructured networks.

**Centralized supervisor-worker patterns** are the most validated approach. A single orchestrator validates outputs before passing them downstream, acting as a quality gate that catches errors before propagation. DeepMind's data shows this reduces amplification from 17.2× to 4.4×. The risk is bottleneck formation at the supervisor, mitigated by giving workers bounded autonomy and escalating only edge cases. Production systems like Anthropic's Claude Code Review and Qodo 2.0 both implement this pattern, with the supervisor serving double duty as aggregator and quality filter.

**Directed acyclic graph (DAG) topologies** outperform unstructured arrangements. The MacNet study (ICLR 2025) found that organizing agents into DAGs yields better results than flat or mesh topologies, with irregular DAGs outperforming regular ones. This aligns with the intuition that code review concerns have natural dependency relationships — a security agent's findings about input validation may inform the correctness agent's analysis of downstream logic.

**Verification checkpoints** inserted at critical junctures catch errors before they compound. The recommended practice is to keep chains under five sequential steps and insert verification agents at key stages. If verification fails, the workflow routes to a human or fallback path — not to a retry of the same chain, which risks amplifying the original error. The BATS framework (DeepMind, November 2025) implements dynamic task trees with self-verification modules, achieving **95% improvement over standard approaches** on the BrowseComp benchmark with 40% lower cost.

A study on resilience under faulty agents (OpenReview, 2025) provides direct evidence for hierarchy's protective effect. The **hierarchical structure exhibited the lowest performance drop at 5.5%**, compared to 10.5% for other structures and **23.7% for flat architectures**. Information boundaries inherent in hierarchy — each layer seeing only what it needs — reduce the hallucination surface area and constrain error propagation through explicit accountability boundaries.

## Production systems resolve conflicts through judge agents and aggregation layers

No production multi-agent code review system relies on agents reaching consensus autonomously. Instead, they use dedicated conflict resolution mechanisms that fall into three categories: judge/aggregator agents, consensus scoring, and conservative refusal patterns.

**Qodo 2.0** (released February 2026) provides the most explicitly documented conflict resolution architecture. Its **Judge Agent** sits atop multiple domain experts and performs four functions: resolving inter-agent conflicts (e.g., a performance agent flagging an optimization that contradicts the architecture agent's preference for clarity), calibrating findings against team-specific priorities, deduplicating observations that multiple experts identify from different angles, and filtering noise — reducing 30 raw observations from five agents into a focused set of actionable findings. The judge weighs tradeoffs rather than surfacing contradictory feedback to developers.

**Anthropic's Claude Code Review** (launched March 2026) dispatches multiple specialized agents in parallel — logic validation, security, regression testing, pattern compliance, performance — then uses a **separate aggregator agent** to collect findings, remove duplicates, filter false positives, and rank remaining issues by severity. Results appear as a single consolidated comment, not contradictory feedback. Internal data shows a **less than 1% false positive rate**, with 84% of large PRs receiving substantive findings. Agents never approve PRs; they categorize findings by severity (must-fix, nit, pre-existing) and leave the merge decision to humans.

**CodeRabbit** takes a conservative refusal approach for ambiguous cases. When resolving merge conflicts, its AI agent explicitly declines if the conflict involves security-critical code — authentication, encryption, access control — or fundamentally incompatible business logic. If any file's resolution is ambiguous, **no commit is created for any files**. This "fail-safe by default" pattern acknowledges that some conflicts genuinely require human judgment.

Academic research on conflict resolution offers a surprising finding. The "Debate or Vote" study (2025, arXiv:2508.17536) conducted systematic comparisons of multi-agent debate versus simple majority voting across seven benchmarks and found that **most performance gains of debate stem from majority voting rather than the debate process itself**. Debate dynamics were characterized as a martingale process that preserves expected success probability over rounds — meaning deliberation doesn't systematically improve beliefs. The MALLM framework (ACL Findings 2025) added nuance: consensus strategies work better for knowledge tasks, while voting works better for reasoning tasks. For code review, which combines both knowledge retrieval (is this a known vulnerability pattern?) and reasoning (does this logic handle edge cases correctly?), a hybrid approach is warranted.

Amazon CodeGuru and GitHub Copilot code review, notably, are **not multi-agent systems** in the contemporary sense. CodeGuru combines complementary analysis layers (ML rules, Bandit for security, Infer for concurrency) with domain partitioning that structurally minimizes contradictions. Copilot runs a single agentic model with tool-calling capabilities rather than dispatching parallel specialists. Both rely on the human developer as the ultimate arbiter.

## Coarse-grained specialization at 5–7 domains outperforms fine-grained decomposition

The evidence strongly favors one "security" agent over separate injection, auth, crypto, and secrets agents. Production systems, academic studies, and theoretical analysis all converge on **5–7 coarse-grained domain agents** as the sweet spot.

**RevAgent** (Li et al., November 2025, arXiv:2511.00517) provides the strongest academic evidence. It decomposes code review into five category-specific "commentator" agents — Refactoring, Bugfix, Testing, Logging, and Documentation — plus a sixth "critic agent" that selects the most relevant findings. This moderate-grained architecture achieved **BLEU +12.90%, ROUGE-L +10.87%, METEOR +6.32%** over the best baselines. The five categories emerged from empirical analysis of real code review comment distributions, not arbitrary splitting.

Production systems independently converge on similar numbers:

- **Diffray**: 9 specialized agents (Security, Performance, Bugs, Architecture, Testing, and others)
- **Claude Code Review**: Multiple parallel agents, each analyzing a different "class of issue"
- **NeoLab**: 6 agents (bug-hunter, security-auditor, code-quality-reviewer, contracts-reviewer, historical-context-reviewer, test-coverage-reviewer)
- **CodeAgent** (academic, arXiv:2402.02172): 5 role-based agents simulating an engineering team
- **Google ADK patterns**: 4 sequential agents (Code Analyzer, Style Checker, Test Runner, Feedback Synthesizer)

The arguments against fine-grained decomposition are compelling. First, **context dilution** — splitting security into sub-agents means each sub-agent lacks the cross-cutting context to see how injection relates to auth or how crypto relates to secrets management. A single security agent can reason about the codebase's overall security posture. Second, **coordination overhead** — each additional agent adds to the 36.9% coordination failure rate documented by the MAST study. Third, the **DeepMind saturation threshold** caps meaningful gains at roughly 4–8 agents. Fourth, **semantic overlap** between fine-grained concerns (injection and auth, crypto and secrets) creates exactly the redundancy that multi-agent architectures should avoid.

The recommended decomposition based on converging evidence uses two tiers. High-value agents that should always be included are **Correctness/Bugs, Security, and Architecture/Design**. Important agents for production systems include **Performance, Testing Adequacy, and Code Quality/Maintainability**. Context-dependent agents — Documentation, Logging/Observability, Compliance — should be added only when domain requirements justify them. **Style and formatting should never be an LLM agent**; deterministic linters (ESLint, Prettier, Ruff) handle this faster, cheaper, and more consistently as a preprocessing step.

The optimal granularity does shift with context. Security-critical domains (fintech, healthcare) may justify promoting security sub-concerns to separate agents. Performance-critical domains (gaming, high-frequency trading) may warrant fine-grained performance decomposition. Languages with strong type systems (Rust, TypeScript) need less from a correctness agent, while dynamic languages (Python, JavaScript) benefit more from bug detection. But even in these cases, the evidence suggests staying under 9 total agents.

## MARG's leader-worker-expert pattern outperforms flat dispatch but at significant cost

MARG (Multi-Agent Review Generation, arXiv:2401.04259, January 2024) is designed for **scientific paper review**, not source code review, but its hierarchical architecture has become a foundational reference pattern. The system uses three agent types: a **Leader** (central coordinator that creates plans and delegates), **Workers** (each holding a chunk of the document under review), and **Experts** (given specialized prompts for specific concerns like experiments, clarity, or impact).

Three independent multi-agent groups run in parallel, each focusing on a different review dimension. Each group has its own leader, workers, and expert. A refinement stage then evaluates, prunes, or improves each comment. In user studies on 10 scientific papers, MARG-S (the specialized variant) produced **2.2× more "good" comments per paper** (3.7 versus 1.7 for the best single-agent baseline) and reduced generic comments from **60% to 29%**. Critically, the specialization — separating review dimensions into distinct groups — was essential; MARG without specialization performed significantly worse.

The tradeoffs between hierarchical and flat architectures are well-quantified. On **parallelizable tasks** (which code review largely is), DeepMind found centralized hierarchical systems improved performance by **+80.9%** over single agents. On sequential tasks, every multi-agent variant degraded performance by 39–70%. LangChain's June 2025 benchmarking found that flat swarm architectures slightly outperformed supervisors across the board due to "telephone" translation overhead — information loss as messages pass through the supervisor layer. However, after optimizations (removing handoff messages, forwarding context directly, better tool naming), supervisor performance improved roughly 50%.

The key advantage of hierarchy is **error containment**. The resilience study finding — 5.5% performance drop under faulty agents for hierarchical versus 23.7% for flat — is the strongest argument for hierarchy in code review, where false positives and error propagation directly impact developer trust. The key disadvantage is **coordination overhead and information loss**. MARG's own failure analysis found that 53% of errors were "missing context" — the leader failing to relay critical information to workers. This telephone problem is inherent to hierarchy.

For code review specifically, the evidence points toward a **shallow two-level hierarchy**: a triage/router agent that analyzes the diff and routes to domain specialists, parallel domain agents that analyze concurrently, and a synthesis/aggregation agent that deduplicates and ranks. This captures hierarchy's error-containment benefits while minimizing the telephone problem. Most production systems (Claude Code Review, Qodo 2.0, diffray) implement exactly this pattern. Deeper hierarchies — three or more levels — add latency and cost with diminishing quality returns.

## Coordination patterns that eliminate redundant work

The SQL query example — where both a bug detector and security reviewer investigate the same code — illustrates the central coordination challenge. Production systems use five primary patterns to prevent this waste, often in combination.

**Domain-scoped partitioning** is the most common and effective approach. Each agent receives a clearly bounded mandate: the security agent analyzes only vulnerabilities, the performance agent only complexity and resource usage, the architecture agent only patterns and coupling. Diffray implements this through "curated context management" — each agent receives only files and context relevant to its domain, so the security agent gets authentication flows while the performance agent gets hot paths. This structural separation makes overlap architecturally difficult rather than relying on agents to avoid it behaviorally.

**Triage-phase routing** prevents overlap before review begins. Diffray's 9-phase pipeline includes a dedicated Triage step (Phase 4) that analyzes file types, content, and dependency relationships to route specific files to specific agents. Claude Code Review dynamically scales agent count based on PR complexity — trivial changes get a lighter pass, large changes receive more specialists. This pre-routing ensures agents examine complementary, not redundant, code regions.

**Post-hoc deduplication through aggregator agents** catches residual overlap. Even with domain partitioning, a buffer overflow finding from the security agent and the same finding framed as a correctness issue by the bug detector can occur. Anthropic's aggregator agent explicitly removes duplicates after parallel analysis. Diffray includes a dedicated "Dedupe" phase (Phase 7) that merges and rescores findings. The Calimero AI Code Reviewer uses **consensus-based scoring** that naturally deduplicates: identical findings from multiple agents are merged and weighted higher rather than presented separately.

**Token duplication rates** across frameworks reveal significant waste. ICLR 2025 Workshop data shows CAMEL duplicating **86% of tokens**, MetaGPT **72%**, and AgentVerse **53%** — meaning multi-agent systems consume 1.5× to 7× more tokens than theoretically necessary. CrewAI mitigates this with a **cross-agent caching layer** that prevents redundant API calls; if one agent already retrieved data, others use the cached result. LangGraph uses unique output keys per agent in shared state, preventing race conditions while enabling downstream aggregation.

**Blackboard architectures** offer a promising alternative to direct agent communication. The LbMAS framework (2025) implements a shared knowledge repository where agents read and write findings rather than communicating directly. It includes a dedicated **cleaner agent** that removes redundant messages and a **conflict-resolver agent** that detects contradictions — addressing both duplication and conflict in the shared state layer. AWS's "Arbiter pattern" extends this with a semantic blackboard enabling dynamic, mid-task coordination based on evolving task state.

A provocative counterpoint comes from the "When Single-Agent with Skills Replace Multi-Agent Systems" paper (2025), which found that compiling multi-agent roles into single-agent skill libraries **reduces token consumption by 54% and latency by 50%** on average. For tasks where coordination overhead exceeds the benefit of specialization, a single agent with domain-specific prompting may outperform an entire multi-agent system. This suggests the right question isn't always "how many agents?" but "does this task genuinely benefit from agent decomposition at all?"

## Conclusion: the architecture that emerges from the evidence

The research converges on a specific, empirically grounded architecture for multi-agent code review. **Five to seven coarse-grained domain agents** (security, correctness, performance, architecture, testing, code quality) operating in a **shallow two-level hierarchy** with a triage router and a deduplication aggregator represents the evidence-based optimum. This keeps agent count below the saturation threshold, uses domain partitioning to minimize overlap, employs hierarchical coordination to contain errors, and maintains cost efficiency through bounded parallelism.

Three design principles should guide implementation. First, **heterogeneity over homogeneity** — using different models, prompts, or tool configurations for each agent delays the saturation ceiling by maximizing non-redundant information channels. Second, **deterministic tools for deterministic tasks** — linters, SAST scanners, and formatters should run as preprocessing, not as LLM agents, reserving expensive inference for genuinely judgment-dependent concerns. Third, **fail-safe by default** — when agents disagree on security-critical findings, the system should escalate to humans rather than resolve autonomously, following CodeRabbit's conservative refusal pattern.

The biggest gap in the literature is the absence of controlled, peer-reviewed benchmarks comparing single-agent versus multi-agent code review on standard metrics like precision, recall, and false positive rates. The DeepMind scaling study provides the closest analog, but code review-specific evaluation remains dominated by commercial claims rather than reproducible science. Until that gap closes, practitioners should treat the 4–6 agent saturation threshold as a strong prior, invest in coordination infrastructure before adding agents, and measure carefully whether each additional agent genuinely improves outcomes after accounting for coordination costs.
