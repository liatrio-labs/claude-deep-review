# Filtering accurate-but-irrelevant findings in automated code review

The core challenge — that **13 of 17 false positives are factually correct observations that aren't real defects** — maps directly onto one of the most well-documented phenomena in software engineering research. Mantyla & Lassenius (2009) found that **75% of code review findings are evolvability concerns, not functional defects**; Bacchelli & Bird (2013) at Microsoft found actual defects comprise only **~12.5% of review comments**. Your 76% ratio (13/17) sits squarely in this empirically validated range. The proposed dual-score approach of adding `confidence_claim_is_defect` alongside accuracy is directionally well-founded — HubSpot, OpenAI, and Snyk have independently converged on similar multi-dimensional architectures — but the specific `min()` aggregation function is suboptimal, and the blind challenger's context limitations create a structural obstacle to reliable significance assessment.

---

## Google's "effective false positive" is the key concept

The most important academic framing comes from Sadowski et al. (2018) at Google, who coined the term **"effective false positive"** — a technically correct finding that developers don't act on. Their definition: "To a tool user, a false positive report is any report that they did not want to see." This redefines the problem away from factual accuracy entirely. Google's Tricorder platform operationalized this by tracking "Not Useful" button clicks per analyzer, disabling any analyzer exceeding **10% effective false positive rate**, and achieving ~5% overall. The platform processes 716 "Please Fix" clicks per day against only 48 "Not Useful" clicks.

Facebook's experience with Infer provides the starkest evidence that accuracy is disconnected from actionability. Distefano et al. (2019) report that when Infer results were filed as bugs (batch mode), **the fix rate was near zero** — despite a technical false positive rate below 20%. When the identical analysis was moved to diff-time code review integration, **fix rates rocketed to over 70%**. Same findings, same accuracy, dramatically different actionability. This demonstrates that actionability is a function of context, timing, and presentation — not correctness.

Imtiaz et al. (2019) studied Coverity adoption across five major open-source projects (Linux, Firefox, Samba, Kodi, Ovirt-engine) and found fix rates of only **27.4% to 49.5%** even in projects actively using the tool. Johnson et al. (2013), interviewing 20 developers, concluded bluntly: "Reporting the factual outputs of the analysis is not sufficient for developers to adopt a static analysis tool." The academic consensus is unambiguous — factual accuracy is necessary but nowhere near sufficient.

## How static analysis tools actually filter for actionability

No mainstream static analysis tool fully automates the accuracy-vs-actionability distinction, but several encode it partially through different mechanisms.

**SonarQube's Security Hotspot vs. Vulnerability distinction** is the clearest example. A Vulnerability means "a problem has been discovered and needs immediate fixing." A Security Hotspot means "a security-sensitive code pattern exists, but impact depends on context — the developer must review." Hotspots go through their own lifecycle (To Review -> Acknowledged -> Safe or Fixed), where marking "Safe" explicitly means "no change necessary." SonarQube also renamed "Won't Fix" to "Accept" in recent versions, reframing the semantics from refusal to acknowledged-risk — explicitly encoding "valid finding, not actionable." However, SonarQube does **not** learn from accumulated Accept/FP resolutions to adjust future analysis. The resolution data stays purely in workflow management.

**Semgrep has the most sophisticated metadata system.** Each security rule carries independent `confidence` (rule precision), `likelihood` (how likely the vulnerability manifests), and `impact` (damage potential) scores. Critically, Semgrep's `subcategory` field directly encodes actionability: **`vuln`** means "specifically looking for an exploitable vulnerability," **`audit`** means "possible vulnerability given other conditions," and **`guardrail`** means policy enforcement. The vuln/audit distinction is precisely the "this IS a defect" vs. "this code pattern EXISTS and might be a defect" separation. Semgrep Assistant, their AI triage layer, goes further — it has triaged ~60% of new SAST findings for customers, with users agreeing with its triage decisions **over 96% of the time**. Most remarkably, Semgrep's "Memories" system learns from past triage decisions and applies them to future findings: one Fortune 500 customer saw a **2.8x improvement** in false positive detection after adding just two memories, with one memory scoped to a single rule identifying 580 additional false positives (up from 18 without it).

**Rust's Clippy** offers the best-designed actionability taxonomy among linters. Its lint groups form an explicit hierarchy: `correctness` (deny by default — "outright wrong or useless code"), `suspicious` (warn — "most likely wrong"), `style` and `complexity` (warn — preference-dependent), `pedantic` (allow — "rather strict, occasional false positives"), and `restriction` (allow — "restricts language features," explicitly not recommended to enable as a group). The design principles state that warn-by-default groups are kept "free from false positives" and that "it is okay if the lint warns about things which people do not feel are worth fixing." This graduated taxonomy maps directly to actionability levels.

**SpotBugs** uses a dual-axis system (Confidence 1-3 x Bug Rank 1-20), where confidence measures tool certainty and rank measures severity, but both are assigned per-rule, not per-finding-in-context. The category system (Correctness, Bad Practice, Style, Performance, Security) functions as an implicit actionability taxonomy — Correctness findings are "definitely a defect" while Style findings are "technically true observations."

## LLM-powered review tools are converging on multi-stage filtering

The most relevant industry precedent is **HubSpot's internal "Sidekick" code reviewer**, which is the only published system explicitly evaluating on separate Accuracy and Actionability criteria. Their architecture uses a Review Agent that generates comments, followed by a Judge Agent that gates each comment on three criteria: **succinctness** (clear and to the point?), **accuracy** (technically correct in context?), and **actionability** (can the suggestion be directly applied?). HubSpot reports this two-stage process was "arguably the single most important factor" in achieving an **80%+ thumbs-up rate** from developers. This is not a commercial product but a well-documented internal system.

**OpenAI's Codex code reviewer** (December 2025) provides the strongest theoretical foundation with an explicit **expected utility formula**: `P(correct) x C_saved - C_human_verification - P(incorrect) x C_false_alarm`. Their key design insight: "Precision is more important for usability than recall" and "comments that are technically correct but more of a stylistic nature may even incur negative utility." The system processes **100k+ external PRs per day** with a 52.7% comment-to-code-change rate, suggesting reasonable significance calibration. Notably, their training-time reward model differs from their deployment model because training prioritizes recall while deployment prioritizes precision — an explicit acknowledgment that finding issues and surfacing actionable issues are different objectives.

**Qodo (formerly CodiumAI)** runs the most documented multi-agent architecture: specialized agents for finding bugs, enforcing standards, assessing risk, and understanding context feed into a Judge Agent that "resolves conflicts, removes duplicates, and filters out low-signal results" based on a "high confidence and relevance threshold." This is structurally closest to the proposed blind challenger approach. **CodeRabbit** takes a different approach with a Verification Agent that generates shell/Python checks (grep, ast-grep) to confirm assumptions before posting — "comments come with receipts." **CodeScene** is philosophically the most aligned with the accuracy-vs-actionability distinction: its hotspot analysis inherently separates "this code has issues" from "this issue matters because this code is frequently changed," with an explicit "No Problem" classification for code where issues exist but aren't worth addressing.

**Snyk** explicitly names **"Completeness, Accuracy, Actionability"** as their three evaluation dimensions for vulnerability curation. **DeepSource** provides the most transparent pipeline documentation, with a multi-stage filtering process (AST analysis -> static processors -> dynamic processors -> relevance engine) and a 5-dimensional PR Report Card (security, reliability, complexity, hygiene, coverage). Several tools including CodeRabbit, Greptile, Sourcery, and Baz learn from developer feedback (thumbs-up/down, reply corrections) to calibrate future reviews, though none publish data on how effectively this feedback loop improves actionability.

## No existing taxonomy explicitly includes an actionability axis

The academic taxonomy landscape is well-developed for categorizing review findings but has a conspicuous gap around actionability. **Mantyla & Lassenius (2009)** proposed the most influential taxonomy, splitting findings into Evolvability Defects (documentation, visual representation, structure — **75% of findings**) and Functional Defects (resource, check, interface, logic, timing — **25% of findings**). **Bacchelli & Bird (2013)** classified 570 Microsoft code review comments into Code Improvements (most common), Defects (~12.5%), Understanding, Social Communication, and Other. **Beller et al. (2014)** corroborated the 75/25 evolvability-to-functional ratio across open-source projects.

**Turzo & Bosu (2023)** produced the most granular academic taxonomy with 18 categories for OpenDev code review comments. **Bosu, Greiler & Bird (2015)** at Microsoft defined "useful" as "led to a code change" and built a Decision Tree Classifier to predict useful vs. not-useful comments — the closest to an operationalized actionability definition. But none of these taxonomies include actionability as an explicit dimension. The closest proxies are the evolvability/functional split (which correlates with but doesn't define actionability) and the useful/not-useful binary label.

The gap is significant: **no published framework explicitly taxonomizes the space between "technically correct finding" and "actionable finding."** Sadowski et al.'s reasons developers ignore warnings (not manifest in practice, too expensive to fix, not understood, not integrated into workflow) provide the closest available framework, but this is a list of barriers rather than a formal taxonomy with categories a machine could apply.

## Adversarial verification can assess accuracy but struggles with significance

The AI Safety via Debate framework (Irving et al., 2018) evaluates "the most true, **useful** information" — explicitly including both truth and utility in its judging criterion. But the paper focuses primarily on truth verification, with "useful" acknowledged but not operationalized. **The debate format does not directly handle the accuracy-vs-significance distinction**; it collapses them into a single judging criterion. The "obfuscated arguments" problem (Barnes & Christiano, 2020) is directly relevant — a dishonest debater can construct technically valid but practically meaningless arguments, analogous to findings that are accurate but not defects.

**CriticGPT** (McAleese et al., 2024) is the most directly relevant system, trained to critique ChatGPT code output with an explicit goal of **producing fewer "nitpicks"** (small, unhelpful complaints). It's preferred over ChatGPT critiques in 63% of cases and uses Force Sampling Beam Search to configure precision-recall tradeoffs. However, CriticGPT still occasionally overstates severity of findings, showing the challenge of significance assessment even with specialized training.

**MAD-Fact** introduces a **"fact importance hierarchy"** that weights claims as "vital," "okay," or "less-important" — the closest direct precedent for separating accuracy from significance in an evaluation pipeline. The related Vital Metrics paper (2024) demonstrates that importance-weighted subclaim evaluation "significantly better distinguishes responses of different quality" compared to treating all claims equally. Multi-agent debate (Du et al., 2023) improves factual accuracy through 3 agents debating over 2 rounds, but focuses exclusively on accuracy, not significance. **PROClaim** reformulates verification as courtroom-style adversarial deliberation with specialized roles (Plaintiff, Defense, Judge, Expert Witness), achieving 81.7% accuracy — the role-based architecture is directly adaptable to separating accuracy and significance assessments.

Evidence on whether LLMs can reliably assess "does this matter?" is **mixed but cautiously optimistic**. LLMs are generally better calibrated on objective tasks than subjective ones. However, calibration probes on hidden states achieve strong calibration for both objective and subjective judgments, and Constitutional AI demonstrates that LLMs can evaluate against multiple competing natural-language criteria when properly prompted. The key limitation: LLMs exhibit systematic biases including **verbosity bias** (favoring findings described more confidently), which could cause overrating of well-articulated but unimportant findings.

## Evaluating the proposed dual-score approach

The proposed fix — adding `confidence_claim_is_defect` alongside factual accuracy and using `min(accuracy, defect_confidence)` as effective score — is **directionally well-founded but has specific design problems**.

**What the literature supports.** The dual-score decomposition is validated by multiple independent convergences: HubSpot's separate Accuracy + Actionability criteria, OpenAI's expected utility formula, Snyk's three-dimensional evaluation, Semgrep's independent confidence/likelihood/impact metadata, and the DMoERM reward model decomposition research. The Vital Metrics paper provides the strongest theoretical backing — importance-weighted evaluation "significantly better distinguishes" responses of different quality. The 75/25 evolvability/functional split across multiple studies confirms that most review findings are not defects, making a defect-confidence dimension empirically justified.

**The `min()` function is too aggressive.** A finding with accuracy 0.95 and defect_confidence 0.4 scores identically (0.4) to one with accuracy 0.4 and defect_confidence 0.95 — but these represent very different situations (a verified non-defect vs. an unverified likely-defect). OpenAI's multiplicative formula `P(correct) x value` provides smoother aggregation. A **product** `accuracy x defect_confidence` or a **weighted geometric mean** would better preserve the distinction between "probably accurate, probably not a defect" and "uncertain accuracy, probably a defect." The product also naturally creates the desired gating behavior: both dimensions must be non-trivial for the score to be meaningful.

**The blind challenger lacks context for significance assessment.** The blind challenge design intentionally strips context to test whether claims hold up independently — this is excellent for accuracy verification but structurally adversarial to significance assessment, which requires understanding how code is used, what users interact with, and what failure modes matter. OpenAI's Codex reviewer showed that **repo-wide tools and code execution significantly improve both precision and significance calibration**. If the challenger agent assesses defect significance with the same limited context it uses for accuracy, it will systematically underperform. The significance assessor likely needs **more context** than the accuracy verifier — inverse to the current design.

**"Would this cause user-facing impact?" is the wrong framing for a blind agent.** This question requires understanding deployment context, user interaction patterns, error handling layers, and business logic that a code-reviewing LLM typically cannot access. A more answerable question: **"Does this code behavior match its apparent intent?"** or **"Could this produce incorrect output under any realistic input?"** These are closer to the factual end of the spectrum and more reliably assessable by an LLM examining code alone.

**Calibration across finding types will be uneven.** LLMs can relatively reliably assess whether a null pointer dereference is a real defect (high confidence) but struggle with whether a missing error log matters (context-dependent). The defect_confidence score will be well-calibrated for some finding categories and poorly calibrated for others, creating inconsistent filtering behavior. Consider **category-specific thresholds** rather than a universal aggregation function.

Five concrete improvements the literature suggests over the baseline proposal:

- **Use `accuracy x defect_confidence` instead of `min()`** — multiplicative aggregation is standard in expected utility frameworks and avoids the cliff-edge behavior of min.
- **Add constitutional significance criteria** to the challenger prompt — explicit natural-language principles like "a finding is significant only if it could cause incorrect behavior visible to end users, affect data integrity, or create a security vulnerability" give the LLM concrete decision criteria rather than asking an open-ended judgment question. Constitutional AI research shows this substantially improves multi-criteria evaluation.
- **Give the significance assessor more context than the accuracy verifier** — provide repository structure, function call graphs, test coverage data, and usage patterns. The current blind design helps accuracy but hurts significance.
- **Implement a feedback loop** — track which findings developers act on and use this to calibrate thresholds over time, as Semgrep's Memories system does. Without feedback, any static threshold will drift.
- **Consider a three-tier output** rather than binary pass/fail: surface high-confidence defects directly, suppress confident non-defects, and flag uncertain findings for human triage with both scores visible. This matches how SonarQube's Hotspot workflow and Semgrep's audit subcategory handle ambiguity.

## Conclusion

The "accurate but not actionable" problem is not novel — it's the dominant challenge in static analysis adoption, documented across two decades of empirical research. What is genuinely novel is attempting to solve it within a multi-agent LLM pipeline rather than through human triage workflows. The literature provides three key insights the proposed design should incorporate. First, **significance assessment requires more context, not less** — the blind challenge design that works for accuracy verification is structurally counterproductive for actionability assessment. Second, **multiplicative aggregation outperforms min-based gating** for combining orthogonal quality dimensions. Third, **the most successful production systems** (Google Tricorder, Semgrep Memories, OpenAI Codex) all rely on feedback loops from developer responses, not just static scoring functions. A dual-score architecture is the right direction — the specific implementation needs context enrichment for the significance dimension, a smoother aggregation function, explicit constitutional criteria for what constitutes a defect, and a mechanism to learn from outcomes over time.
