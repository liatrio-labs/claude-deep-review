# How production AI code reviewers handle pre-existing technical debt

**No production AI code review tool has fully solved the pre-existing debt problem.** The industry has converged on a spectrum of mechanisms — from simple diff-scoping (review only changed lines) to sophisticated baseline comparison algorithms — but each approach makes different tradeoffs between noise and coverage. SonarQube's deterministic line-hash matching algorithm remains the gold standard for separating new from pre-existing issues, while most AI-native tools like CodeRabbit, Qodo, and Ellipsis rely primarily on diff-scoping as an implicit filter and lack any formal baseline comparison. Anthropic's Claude Code Review is the notable exception among AI-native tools: it explicitly filters "pre-existing issues not introduced in PR" using git blame, multi-agent verification, and confidence scoring. The consistency-vs-correctness tradeoff — whether to flag new code that follows existing bad patterns — is handled through manual configuration everywhere, with no tool automatically detecting established codebase conventions.

## The decision boundary: when does pre-existing code become the PR's problem?

The core question — when a PR's new code touches or flows into pre-existing buggy code, should the reviewer flag it? — reveals a fundamental architectural split across tools.

**SonarQube uses deterministic matching, not AI judgment.** Its algorithm compares every issue from the current analysis against the previous analysis using three match paths: (1) same rule + same line content hash (whitespace-excluded), (2) same rule + same line number + same message, or (3) same rule + same block position after detecting code movement within a file. If a current issue matches a previous one by any path, it's classified as pre-existing and excluded from the PR report. Only unmatched issues are flagged as new. This means if a PR touches a function with 5 pre-existing issues and introduces 1 new one, SonarQube analyzes the full file, runs the matching algorithm, and **reports only the 1 genuinely new issue** in the PR decoration. The 5 pre-existing issues literally cannot fail a "Sonar way" quality gate on a PR.

**Codacy takes a simpler line-level approach.** Issues detected on lines changed by the PR are classified as "new." Issues on unchanged lines of modified files are labeled "potential" — clearly separated from definitive new findings. This is more conservative than SonarQube's tracking across refactors but avoids the complexity of cross-analysis issue databases.

**AI-native tools mostly rely on diff-scoping as an implicit filter.** CodeRabbit, Qodo/PR-Agent, Sourcery, and Ellipsis all analyze the PR diff as their primary input and expand context outward. None maintain a baseline issue database or run comparison algorithms. This means their "decision boundary" is essentially the AI model's contextual judgment about whether a finding is relevant to the current change — not a deterministic classification. CodeRabbit's documentation describes its scope as "changed lines within a pull request, which helps reduce noise," while also pulling in related files via its Codegraph (callers, callees, shared types, configuration files). Qodo's open-source PR-Agent uses `extend_patch()` to add **3 extra lines before and 1 after** each diff hunk, plus dynamic upward extension to find enclosing function/class declarations — but the AI receives this surrounding context without any annotation distinguishing "your code" from "pre-existing code."

**Anthropic's Claude Code Review deliberately crosses the boundary.** Unlike diff-only tools, Claude Code Review explicitly surfaces pre-existing issues when they're contextually relevant to the PR. The canonical example: during review of a TrueNAS PR, it found a pre-existing type mismatch in adjacent code that was silently wiping an encryption key cache on every sync — **a bug unrelated to the PR change itself**, exposed because the system analyzed full codebase context. Its open-source plugin uses four parallel agents, including one that explicitly uses `git blame` and history for context-based issues, while another scans specifically for "obvious bugs in changes." The system then applies a **confidence score from 0-100** (threshold: 80) and explicitly filters out "pre-existing issues not introduced in PR" along with "pedantic nitpicks" and "issues linters will catch." A verification phase forces agents to **attempt to disprove each finding** before surfacing it.

## How full-codebase tools avoid flooding PRs with debt findings

Tools that analyze beyond the diff face a fundamental tension: broader context catches more real bugs but risks drowning developers in pre-existing noise. Production systems use three distinct strategies.

**Strategy 1: Analyze everything, report only new (SonarQube model).** SonarQube analyzes full files but gates reporting through its matching algorithm. For PRs, new code is always defined as "code that has changed compared to the target branch," overriding any project-level settings. SonarQube also uses **git blame for issue backdating** — when a new rule or upgrade detects a pre-existing issue, it backdates the issue to the date of the last commit on that line (from SCM data), ensuring it's classified as old code rather than new. During PR analysis, "new issues on old code are NOT reported since only new code issues are reported." This is the most mature approach but requires maintaining an issue database across analyses.

**Strategy 2: Multi-stage confidence filtering (Anthropic/Ellipsis model).** Claude Code Review and Ellipsis both generate findings broadly, then aggressively filter before posting. Ellipsis runs a **four-stage pipeline**: deduplication filter (removes similar comments), confidence filter (threshold-based scoring), logical correctness/hallucination filter (uses attached evidence to detect false positives), and comment editing (adjusts line numbers). Filtered comments and reasoning are included in the output for transparency. Claude Code Review adds a unique **verification/disproval step** where agents challenge their own output. Per one analysis: "Most AI reviewers post every finding they generate. Claude Code Review forces agents to challenge their own output before surfacing it."

**Strategy 3: Learn from feedback over time (Greptile/CodeRabbit model).** Greptile reports ~85% actionable findings initially, improving over 2-3 weeks as teams provide thumbs-up/down feedback. The founder explicitly stated: "There is no universal understanding of nitpickiness. It is why we have it learn every new customer's ways from scratch." CodeRabbit's learnings system lets teams teach preferences via natural language comments on any PR, scoped as local (repo), global (org), or auto. Ellipsis embeds user feedback into similarity search — when a new finding resembles a previously downvoted one, it's suppressed with "near-immediate reflection" and no retraining delay.

## GitHub Code Scanning's fingerprint-based mechanism

GitHub's code scanning (primarily CodeQL) uses a deterministic, non-AI approach that represents the industry's most clearly documented baseline comparison system.

**The core mechanism changed significantly in March 2023.** Before, code scanning "presented all alerts unique to the pull request branch, even if they were unrelated to the code changes." After the change, **alerts appear on a PR only if all lines of code identified by the alert exist in the PR diff**. This is a strict diff-intersection rule — if an alert's source location falls outside the changed lines, it won't appear as a PR annotation even if it's a genuine new finding on the branch.

**SARIF fingerprinting prevents duplicate alerts across runs.** GitHub uses the `partialFingerprints` property from the OASIS SARIF standard. Each alert is identified by its `ruleId`, `filepath`, and computed fingerprint from the code content. When a new analysis runs, alerts with matching fingerprints are recognized as the same issue rather than creating duplicates. The reference implementation lives at `github/codeql-action/blob/main/src/fingerprints.ts`. Alerts that exist on the default branch show their current status (open/fixed/dismissed), while alerts that don't exist on the default branch appear labeled as "in pull request" or "in branch."

**A key limitation exists for data-flow alerts.** CodeQL performs formal taint-tracking analysis that traces data from sources to sinks across methods and files. However, for these alerts to appear on a PR, both the source and sink locations must fall within the diff. **Pre-existing data-flow paths that gain a new source from the PR may not be shown if the sink is in pre-existing code outside the diff.** This is a known gap in the diff-intersection approach — precisely the scenario where new code feeds data into existing buggy code.

Among AI review tools, **no tool implements GitHub's exact fingerprint-based approach**. The closest parallel is SonarQube's line-hash matching algorithm, which predates GitHub's implementation. JetBrains' Qodana is the only AI-adjacent tool with an explicit baseline system: run an initial analysis, set the result as a baseline file, and subsequent scans report only new issues — "preventing the wall of legacy issues problem."

## Noise thresholds and adaptive strictness in high-debt codebases

No production tool implements a true "tech debt budget" that automatically suppresses cross-file findings when a repo exceeds a threshold. However, several tools approximate this concept through different mechanisms.

**CodeScene comes closest to a genuine tech debt budget.** It assigns a Code Health score (1-10) to every file and focuses enforcement exclusively on "hotspots" — files with both high development activity and declining health. The philosophy: "An organization can live with a certain amount of code quality issues and technical debt, as long as those issues are in the right parts of the code — stable and well-understood areas." Three PR gate profiles let teams choose enforcement levels:

- **Recommended**: Enforces critical rules on new code, prevents health decline in hotspots only
- **Full enforcement**: Fails checks for any file where health declines
- **Elevate Code Health**: Actively enforces improvement goals

CodeScene's peer-reviewed research found unhealthy code has **15x more defects** and **2x slower development**, which drives their hotspot-focused approach. Teams can set goals per hotspot and explicitly accept temporary debt: "Maybe the current feature is so important that it might be worth taking on some technical debt? You just redefine your goal. The point is that such decisions are now transparent and intentional."

**SonarQube's "fudge factor" suppresses noise on small changes.** Quality gate conditions on duplication and coverage are ignored until the number of new lines reaches at least 20. This prevents a 3-line fix from failing the gate because it touched a poorly-covered file. SonarQube Server 2025+ also introduces a **sandbox feature** where issues introduced by platform upgrades (new rules, improved analyzers) are automatically sandboxed — they don't impact quality gates or ratings and can be triaged at the team's pace.

**Per-tool noise controls vary significantly.** Qodo's PR-Agent defaults to **`num_max_findings=3`** per review, a hard cap. CodeRabbit offers `profile: "chill"` (lighter feedback) vs. `profile: "assertive"` (more thorough). Ellipsis provides a configurable confidence threshold and a "Quiet Mode" that posts comments only when findings exceed confidence requirements. Cursor Bugbot claims **~90% actionable** findings by flagging only high-impact issues and avoiding style/formatting entirely. Graphite Agent reports a **<3% false positive rate**. GitHub Copilot Code Review posts nothing at all in **29% of reviews** — choosing silence over noise.

**Real-world noise data is instructive.** One team tracked 28 CodeRabbit PRs and categorized findings: quality improvements 35%, nitpicking 21%, useless/noise 15%, wrong assumptions 13%, thoughtful 13%, security/critical 3%. Another team reported that after a week and a half, teammates were "auto-dismissing CodeRabbit the way they auto-dismiss Dependabot alerts" due to notification volume — each inline comment generates a separate GitHub notification. Riskified built a custom review agent using Claude Code specifically because "standard AI review tools were too noisy for our team," implementing "dynamic skills and active noise cancellation" with negative constraints and a multi-agent 2-of-4 filter criteria.

## The consistency-vs-correctness tradeoff remains manually configured

When new code follows a bad pattern that already exists elsewhere in the codebase, should the reviewer flag it (it's new bad code) or skip it (it's consistent with the codebase)? **No production tool automatically detects this scenario.** The tradeoff is handled entirely through manual configuration.

**Configuration-file approaches dominate.** GitHub Copilot reads `.github/copilot-instructions.md` where teams document patterns with explicit anti-pattern/correct-pattern comparisons. Claude Code Review uses `CLAUDE.md` for project context and `REVIEW.md` for review-specific scope — teams explicitly specify what to ignore and what to prioritize. CodeRabbit's `path_instructions` allow different review standards per directory (e.g., `"src/old-module/**": "Legacy code - only flag security issues"`). One practitioner noted: "Same model. Same diff. Completely different review" — project context files fundamentally transform the consistency/correctness balance.

**Learning-from-feedback approaches offer implicit handling.** Qodo's auto-best-practices feature tracks accepted suggestions and maintains a best-practices wiki specific to the codebase — if teams consistently accept code following existing patterns, the system learns those patterns as acceptable. CodeRabbit's semantic index surfaces "parallel implementations to align with," which may actually **encourage** following existing patterns, whether good or bad. Greptile catches "inconsistencies with existing patterns" through its full codebase graph and learns from thumbs-up/down reactions what the team considers important.

**The tension is real and unresolved.** CodeRabbit's study of 470 PRs found "AI-produced code often looks consistent but violates local patterns around naming, clarity, and structure." The single biggest quality gap was readability — AI code drifts toward "generic defaults" rather than repo idioms. This suggests that flagging pattern violations is valuable even when the codebase is inconsistent, but the HackerNews discussion on this topic reveals deep disagreement: one commenter noted "A good code reviewer knows how to drop all the things that irk them and hyperfocus on what matters, if there's a functional issue with the code."

## The practical implementation landscape, summarized

The mechanisms used by production systems fall into a clear hierarchy of sophistication:

| Mechanism | Tools using it | Effectiveness for pre-existing debt |
|---|---|---|
| **Deterministic baseline matching** (line hash + rule + message comparison against previous analysis) | SonarQube, Qodana | Highest — formally separates new from old with code-movement tracking |
| **Diff-intersection** (alert shown only if all identified lines are in the PR diff) | GitHub Code Scanning/CodeQL | High — deterministic but misses cross-boundary data-flow issues |
| **Multi-agent confidence filtering** (generate broadly, verify/disprove, threshold at ≥80) | Claude Code Review | Medium-high — catches adjacent bugs but relies on AI judgment |
| **Multi-stage pipeline filtering** (dedup → confidence → hallucination check → edit) | Ellipsis | Medium — reduces noise systematically but no formal new/old distinction |
| **Diff-scoping + expanded context** (review the diff, use repo for understanding) | CodeRabbit, Qodo, Sourcery, Copilot | Medium — implicit noise reduction via scope, but no formal baseline |
| **Hotspot-aware enforcement** (enforce only on frequently-changed declining code) | CodeScene | Medium — pragmatic for debt-heavy repos but doesn't track individual issues |
| **Feedback learning** (learn from thumbs-up/down over time) | Greptile, Ellipsis, CodeRabbit | Low initially, improves — requires 2-4 weeks of team investment |

## Conclusion

The gap between what teams need and what tools provide is substantial. Teams working in high-debt codebases need **deterministic new-vs-old classification** — and only SonarQube, Qodana, and GitHub Code Scanning deliver this through baseline comparison algorithms. Every AI-native code review tool relies on some combination of diff-scoping and confidence filtering, which reduces noise but cannot guarantee that a pre-existing issue won't be flagged or that a new issue at the boundary won't be missed.

The most promising architectural direction is Claude Code Review's approach of deliberate full-context analysis paired with aggressive multi-agent verification and explicit pre-existing-issue filtering. But the practical reality is that most teams running AI code review in high-debt codebases end up in one of two configurations: either pairing an AI reviewer (CodeRabbit, Qodo, Copilot) with a deterministic SAST tool (SonarQube) for formal baseline gating, or investing 2-4 weeks in feedback training and configuration tuning to reduce noise to acceptable levels. The "tech debt budget" concept — automatically adapting review strictness to a codebase's existing quality — remains largely aspirational, with CodeScene's hotspot-aware enforcement being the closest production implementation. And the consistency-vs-correctness tradeoff is solved nowhere automatically — it remains a team-level decision encoded in configuration files.
