# The silent failure: how agents lose work at the output token cliff

**Most production agent systems don't manage output token budgets at all — and when agents hit output limits, the dominant behavior is silent failure.** Across a survey of 15+ open-source and commercial agent frameworks, only one (PR-Agent) implements explicit output token reservation, and only one (Aider) has a production-grade continuation mechanism. The rest either ignore the problem entirely or address only input context management, leaving a systematic gap where agents complete expensive investigation — tool calls, file reads, reasoning chains — then lose all results when output generation hits a hard token wall. This failure mode is under-measured in benchmarks but increasingly visible in production, where GitHub issues across Claude Code, AutoGen, Agno, Gemini CLI, and others document the exact same pattern: truncated output, no warning, lost work.

The problem is structurally distinct from wrong answers. An agent that produces the wrong code patch wastes compute but gives you signal about what went wrong. An agent that does 20 minutes of correct investigation then truncates mid-JSON gives you nothing — and costs the same. As agent tasks grow more complex and multi-step, this failure mode is becoming more frequent and more expensive.

## Only one system reserves output tokens; the rest manage input and hope

The most striking finding across all surveyed systems is the **asymmetry between input and output token management**. Every major framework invests heavily in input context management — progressive summarization, history condensation, context window monitoring — but virtually none proactively manage the output side.

**PR-Agent is the sole exception.** It implements a dual-threshold output buffer reservation system with `OUTPUT_BUFFER_TOKENS_SOFT_THRESHOLD` and `OUTPUT_BUFFER_TOKENS_HARD_THRESHOLD` constants. When composing prompts from code patches, it tracks running token count and stops adding content when approaching `max_tokens_model - SOFT_THRESHOLD`, deferring remaining files to a secondary pass. At the hard threshold, files are skipped entirely. This operates at prompt composition time — ensuring enough headroom for the model's response — rather than during generation itself, but it's the closest any production system comes to output reservation.

**Aider takes a recovery-first approach** with its "infinite output" mechanism. When it detects `finish_reason: "length"`, it initiates a new LLM request with the partial response prefilled as the assistant's beginning, prompting the model to continue where it left off. This works well for code edits (where partial output is mechanically continuable) but would fail for structured JSON that must be syntactically complete. Only models supporting assistant prefilling (Claude, DeepSeek, Mistral) can use this technique.

**OpenHands has the most sophisticated input-side solution** through its Condenser architecture — a pluggable system offering `LLMCondenser` (LLM-based summarization), `RollingCondenser` (maintaining rolling summaries), and token-based triggers that fire condensation when token count exceeds thresholds. This achieves roughly **2x cost reduction** and prevents context window overflow, but does nothing about output truncation. A telling bug: when the rolling condenser's own output exceeded model limits, it entered an infinite loop — demonstrating how context/output limit handling can cascade into worse failures when implemented incorrectly.

**Claude Code relies on model-level context awareness** — Anthropic has trained Claude 4.5+ models to track their remaining context window and manage output accordingly. This is the most architecturally elegant approach (the model itself knows when to wrap up) but is opaque, uncontrollable, and limited to Anthropic's newest models. Claude Code's subagent processes are hardcoded to a **32,000 token output limit** that doesn't respect the `CLAUDE_CODE_MAX_OUTPUT_TOKENS` environment variable, a gap documented in multiple GitHub issues where agents processing 1,200+ line synthesis tasks fail entirely at the 32K wall.

**SWE-Agent, Cursor, and CodeRabbit show no evidence of output budget management.** SWE-Agent retrieves max output tokens from litellm's model registry (which itself has been a source of bugs — reporting Claude 3.7 Sonnet as supporting 128K output when the actual limit was 64K) but implements no reservation, checkpointing, or continuation logic.

## Silent truncation is the dominant failure pattern in practice

GitHub issue analysis across frameworks reveals a consistent and concerning pattern: **agents hit output limits silently, with no warning, no recovery, and no user notification.** This was documented in at least eight separate frameworks.

The most instructive case is **Agno (Issue #6910)**: when input context consumes the token budget, both OpenAI Chat Completions (`finish_reason: "length"` with `content: null`) and the Responses API (`status: "incomplete"` with empty `output_text`) return truncated responses that are silently parsed into empty `ModelResponse` objects. The agent then loops endlessly or produces downstream errors. The root cause is simple — `_parse_provider_response` never checks for truncation signals. A similar pattern appears in Agent Zero (Issue #1162), where truncation from providers like DeepInfra (capped at 16,384 tokens) is silently accepted with no log warning. In Gemini CLI (Issue #2104), the codebase doesn't check the `finishReason` field at all — users think the CLI is broken when responses stop mid-sentence.

**Incomplete tool call JSON** is a particularly destructive variant. When an agent generates a `tool_use` or `function_call` block that exceeds `max_tokens`, the JSON parameters are truncated mid-string, producing invalid JSON that fails tool execution. Anthropic's own documentation warns: "there is no guarantee that the resulting stream will complete in a valid JSON string... if the stop reason max_tokens is reached, the stream may end midway through a parameter." In AutoGen (Issue #7115), when Anthropic Sonnet output exceeds max_tokens and splits into parts, subsequent agents receive empty input for their function tool calls — the intended content is simply lost.

**Hardcoded limits mismatched with model capabilities** compound the problem. AnythingLLM hardcodes `max_tokens: 4096` for Anthropic models that support 8,192–128,000+ tokens. Google ADK hardcodes Claude at 4,096. LangChain's `ChatAnthropic` defaults to 1,024 despite documentation claiming `None`. These legacy defaults, appropriate for 2023-era models, silently truncate modern models well below their actual capability. The pattern is so common it suggests a systemic underinvestment in keeping token configurations current.

A Claude Code feature request (Issue #8775) articulates the ideal solution: **real-time output token budget monitoring with graduated handoff** — at 80% capacity, begin summarizing progress; at 95%, force handoff with full state preservation. No framework implements this today.

## Structured generation frameworks cannot prevent truncation, only detect it

A thorough survey of constrained decoding and structured generation frameworks — Outlines, Instructor, LMQL, Guidance, SGLang/XGrammar — reveals a fundamental limitation: **constrained decoding guarantees each token is valid given prior tokens, but cannot guarantee the output will be complete before hitting the token limit.** The grammar tracks where you are in the structure but cannot force early termination with valid closing tokens.

All major APIs exhibit the same behavior. OpenAI's documentation states explicitly: "The model can fail to follow the schema if the generation reaches max_tokens or another stop condition before finishing." Anthropic's structured outputs (GA on Claude 4.5+) produce truncated, invalid JSON when `stop_reason` is `max_tokens`. There is no graceful degradation — the output is simply broken.

The most promising patterns for incremental structured emission are:

- **Instructor's `Partial[T]` wrapper** streams partially-filled Pydantic objects, yielding fields as they're generated. Its `create_iterable` method streams arrays, yielding each complete element — the clearest "emit-as-you-go" pattern for multi-item extraction. However, validators cannot run on partial data, and a regression (Issue #665) shows this feature has reliability challenges.
- **LMQL's per-variable generation** is inherently incremental — variables generated in a loop are immediately available with per-variable token constraints (`len(TOKENS(VAR)) < N`), providing fine-grained control that naturally resists global token limit exhaustion.
- **Guidance's fast-forward tokens** reduce the number of generated tokens needed for structured output by injecting fixed tokens (like JSON scaffolding) without consuming generation budget, effectively expanding the output budget.
- **XGrammar-2** (January 2025) adds dynamic structured generation for agentic tasks with `TagDispatch` for tool-calling format switching, achieving **6x compilation speedup** — critical for agentic workloads where schemas change per-request.

The **EMNLP 2024 paper "Let Me Speak Freely?"** provides important empirical evidence: rigid format constraints hurt performance on reasoning-heavy tasks (math, multi-hop QA) but help classification tasks. The recommended pattern is a hybrid "scratchpad + final_answer" approach — free-form reasoning followed by a validated structured field. This directly addresses the tension between thorough investigation and structured output, though it doesn't solve the truncation problem.

A clear **research gap** exists for truncation-safe structured generation — algorithms that detect approaching token limits and force the grammar into a valid terminal state (closing all open brackets, terminating strings). The closest work is **IterGen (ICLR 2025)**, which enables backtracking and re-generation tied to grammar symbols with KV-cache reuse, but it addresses semantic correction rather than token-limit-aware termination.

## Recovery strategies exist but are ad hoc and framework-specific

The most widely deployed recovery strategy is **continuation prompting**: detect `finish_reason: "length"`, append the truncated response as an assistant message, send a "Continue from where you left off" user message, and accumulate results. This pattern appears in blog posts, LangGraph examples, and Aider's production code. It works for free-form text and code but breaks for structured output that must be syntactically complete in a single pass.

**LangGraph's architecture provides the best primitives** for building truncation recovery without having it built in. Its `StateGraph` with conditional edges on `finish_reason` enables state machine patterns where truncation triggers continuation nodes. Built-in checkpointing persists workflow state after each node, enabling resume from the last valid checkpoint. Per-node retry policies (`RetryPolicy(retry_on=ValueError, max_attempts=2)`) handle transient failures. But these must be manually wired up — LangGraph doesn't detect or recover from truncation automatically.

**DSPy's assertion system** is the most programmatic approach. `dspy.Assert` and `dspy.Suggest` define computational constraints on module outputs; when constraints fail, DSPy backtracks and retries with updated prompts including past failures and error messages. The optimization pipeline (BootstrapFewShot, MIPROv2) indirectly selects for outputs that don't truncate by filtering traces to keep only those passing metrics. This is oriented toward semantic correctness rather than truncation specifically, but the mechanism generalizes.

**Separate structuring steps** are emerging as a practical pattern. The n8n workflow platform explicitly recommends: "Use a separate LLM-chain to receive the data from the agent and parse it. This leads to better, more consistent results than parsing directly in the agent workflow." Google ADK supports sub-agents for output formatting. Mastra uses a dedicated "structuring model" when the main agent isn't proficient at structured output. This architecture naturally limits truncation risk — the structuring call has minimal input context and focuses entirely on formatting already-gathered results.

**Anthropic's automatic tool call clearing** (`clear_tool_uses_20250919` beta) represents a provider-level solution: Claude automatically clears old tool use results as conversations approach token limits, freeing space for new output. Combined with **token-efficient tools** (saving an average of **14% in output tokens, up to 70%**), this reduces the likelihood of hitting output limits during tool-heavy agentic workflows.

## Empirical data shows context exhaustion is a top failure mode, but output truncation is unmeasured

No agent benchmark specifically tracks output truncation rate as a metric. The closest proxy data comes from failure mode analyses that identify context-related failures without distinguishing input overflow from output exhaustion.

**SWE-Bench Pro** (Scale AI, 2025) provides the most detailed breakdown: **context overflow accounts for 35.6% of Claude Sonnet 4 failures** — the dominant failure mode for the strongest models, exceeding even semantic/algorithmic errors (35.9% for Opus 4.1). Syntax errors are frequently attributed to "context truncation effects." This aligns with the finding that coding agents spend **60%+ of tokens on context-finding** rather than problem-solving.

**SWE-Next** (2025) analyzed 80,036 trajectories and found **87.4% of failures end by hitting the step budget** rather than crashing — the agent runs out of iterations, not tokens. But this masks output truncation, which would appear as a step that produces no usable result.

The withdrawn ICLR 2026 submission "How Do Coding Agents Spend Your Money?" provides the first empirical analysis of agent token consumption patterns using OpenHands on SWE-bench. Key findings: task token consumption varies by **up to 10x** on the same task across runs, input tokens dominate cost even with caching, and predicting total consumption achieves Pearson's r < 0.15. Output token prediction shows only "weak-to-moderate correlation" — suggesting output length is highly unpredictable, making static budget allocation unreliable.

**Failed agent trajectories are consistently longer than successful ones**: SWE-agent failures are **12.6–18.5% longer**, OpenHands failures are **31–82.5% longer**, and Prometheus failures are **50–57% longer** (ASE 2025). Longer trajectories accumulate more context, increasing both the probability of hitting context limits and the cost of wasted work when output truncation occurs.

The economic cost is substantial but unquantified in isolation. A failed SWE-agent run costs a mean of **$2.52** versus $1.21 for success. At Anthropic's >200K token pricing tier (which charges 2x), a single wasted request can cost $1–2+. Re-dispatch requires resending most of the accumulated context, roughly doubling the cost of the failed attempt. Users report Claude Code sessions consuming **€16+ per day** and exhausting $200/month plans in under 20 minutes during complex agentic tasks.

Epoch AI's SWE-Bench Verified scaffold sets token limits at **2M uncached read/write tokens** and notes "we have not observed models hitting these high token limits in practice, other than in pathological cases such as getting stuck in a loop." This suggests that with sufficiently generous limits, output truncation is rare in benchmark settings — but production deployments operate under tighter cost constraints where generous limits are impractical.

## Design patterns for truncation-robust agent output remain nascent

Engineering practices for designing agent output formats robust to truncation are not yet codified into standard patterns, but several emerging approaches show promise.

**Tool calls as a separate structured channel** is the most production-ready pattern. By emitting structured results as tool call arguments rather than text output, agents use a mechanism that's parsed separately and typically much smaller than full text output. OpenAI distinguishes the use cases: function calling for system-facing structured data, `response_format` for user-facing structured text. LangChain's `ToolStrategy` implements this by having models emit structured data as tool calls. The key advantage: each tool call is independently parseable, so even if the overall response truncates, previously-emitted tool calls remain valid.

**NDJSON (newline-delimited JSON)** is a simple but effective pattern for incremental emission: each line is a complete, independently-valid JSON object. Community discussion suggests this is "much simpler to implement" than complex schemas for LLM output, and truncation only loses the final incomplete line. The Vercel AI SDK's `Output.array()` with `elementStream` provides a production-ready implementation that yields each complete array element during streaming.

**Periodic structured checkpoints** — requiring agents to emit structured state at regular intervals — are supported architecturally by LangGraph's built-in state persistence but are not standard practice. The recommended pattern stores structured meaning (completed steps, intermediate results, pending work) rather than raw LLM text, enabling recomputation of downstream steps from the last valid checkpoint.

**Proactive token monitoring middleware** is the least-implemented but most-needed pattern. One blog post describes "token monitoring middleware that automatically triggers a summarization or truncation event when the context hits a percentage of the model's limit." The Claude Code feature request for graduated handoff (80% → begin summarizing, 95% → force handoff) articulates this pattern precisely. The ACL Findings 2025 paper "Token-Budget-Aware LLM Reasoning" demonstrates that budget-aware approaches achieve **1–6% accuracy gains and 30–70% token compression** versus naïve truncation.

## Conclusion

The output truncation problem sits at an uncomfortable intersection: it's too rare to dominate benchmark metrics but too costly to ignore in production. The core tension — agents need freedom to investigate thoroughly, but must produce structured output before hitting token walls — remains fundamentally unresolved by current tooling.

Three concrete gaps demand attention. First, **no constrained decoding engine can force a valid terminal state when approaching token limits** — this is the most impactful research opportunity, requiring grammars that can "early-close" with valid output. Second, **no agent framework implements real-time output budget monitoring** with graduated responses (summarize → checkpoint → handoff) — the closest proposal exists only as a GitHub feature request. Third, **no benchmark measures output truncation frequency**, making the problem invisible to the research community even as production deployments accumulate silent failures.

The most practical near-term mitigations combine multiple existing techniques: use tool calls for structured data (separate channel), design schemas as arrays of independently-valid items (NDJSON pattern), check `finish_reason`/`stop_reason` and implement continuation logic (recovery), and run structuring as a separate lightweight LLM call after investigation (decomposition). These don't solve the underlying problem but reduce its blast radius from "all work lost" to "graceful degradation with partial results."
