# Incremental AI code review is an unsolved engineering problem

**Most production AI code review tools handle incremental review poorly or not at all.** Only CodeRabbit implements a genuine incremental mechanism — persisting state in hidden PR comments and reviewing only changed files. GitHub Copilot explicitly acknowledges it duplicates comments across re-reviews. Greptile runs fully independent passes. Qodo/PR-Agent has incremental code paths but removed them from documentation. Anthropic's Claude Code Review auto-resolves fixed issues but appears to re-analyze the full diff each time. The gap between what traditional incremental static analysis achieves (Infer's compositional summaries, SonarQube's "new code period") and what LLM-based reviewers deliver is enormous — and it's the primary driver of the **60–80% noise rates** developers report. Building a high-quality incremental AI review system requires solving three hard problems simultaneously: comment lifecycle management against platform API constraints, context persistence without server-side state, and the fundamental tradeoff between catching cross-cutting issues and avoiding notification fatigue.

## CodeRabbit is the only tool with true incremental review architecture

CodeRabbit's approach is the most technically sophisticated and best-documented system for incremental PR review. Its core mechanism stores all review state within the pull request itself — not on CodeRabbit's servers — using hidden HTML comment tags embedded in PR comments. Co-founder Harjot Gill described the design: "For each commit, we summarize diff for each file. Then, we create a summary of summaries, which is incrementally updated as further commits are made on a pull request. This summary of summaries is saved, hidden inside a comment on a pull request."

The hidden comments use parseable HTML tags like `<!-- This is an auto-generated comment: summarize by coderabbit.ai -->` that render invisibly in GitHub's markdown but allow CodeRabbit to locate and update its state. The summary uses a **rolling update approach** — each new commit's per-file summaries are folded into the existing summary rather than replacing it. This design was driven by privacy: "We had to design a system that stores all state within the pull request itself and not in our service for maximum privacy."

CodeRabbit distinguishes between two review modes. A **full review** (`@coderabbitai full review`) disregards all previous comments and analyzes the entire PR diff. An **incremental review** (triggered automatically on each push, or via `@coderabbitai review`) considers all comments since the last full review and generates feedback on only the new changes. The open-source version confirms: "Incremental reviews save on OpenAI costs and reduce noise by tracking changed files between commits and the base of the pull request." The system is triggered by GitHub's `pull_request` webhook with the `synchronize` action type, which fires when new commits are pushed to a PR branch.

The multi-LLM pipeline uses a **tiered model strategy**: lightweight models (historically gpt-3.5-turbo) handle summarization and triage, while heavier models (gpt-4 class) perform actual code review. Simpler models act as a "triage filter that identifies the changes that need to be thoroughly reviewed by more complex models." The diff format is custom — side-by-side with line number annotations rather than unified diff — because "this is similar to how humans want to look at diffs." The system integrates **40+ static analyzers and SAST tools** alongside LLM analysis, with specialized parallel agents (Review, Verification, Chat, Pre-Merge Checks, Finishing Touches) running in a sandboxed cloud environment with the full repository cloned.

A notable architectural limitation: CodeRabbit does **not** automatically resolve outdated comments. The `@coderabbitai resolve` command exists specifically because old comments accumulate and require manual intervention. The walkthrough summary comment is edited in-place, but individual review comments on specific lines are posted as new comments for each incremental pass. Comments on lines that moved due to rebases are handled by GitHub's native "outdated" marking, not by CodeRabbit's own tracking.

## Qodo/PR-Agent defaults to full re-review with deprecated incremental code paths

PR-Agent's open-source codebase reveals that **full diff re-review is the default behavior**. Each `/review` invocation calls `get_pr_diff()` in `pr_agent/algo/pr_processing.py`, which fetches all changed files via `GitProvider.get_diff_files()`, returning `FilePatchInfo` objects for the complete diff between the PR branch and target branch. There is no automatic detection of "what changed since last review."

An opt-in incremental mode exists via `/review -i` (or `--incremental`), implemented in `pr_agent/tools/pr_reviewer.py`. The `PRReviewer.__init__()` method creates an `incremental` state object tracking `is_incremental`, `first_new_commit_sha`, and `last_seen_commit`. When active, it filters to `git_provider.unreviewed_files_set`. Two configurable thresholds gate the feature: `minimal_commits_for_incremental_review` and `minimal_minutes_for_incremental_review`. However, **this feature was removed from documentation in release v0.27** (PR #1571: "Docs: fix static code and remove Incremental PR review"). The code paths remain functional but appear deprecated for the open-source version, likely migrated to the commercial Qodo Merge product.

PR-Agent's **comment management relies on persistent comment editing**. With `persistent_comment = true` (the default), the `publish_persistent_comment()` method in `github_provider.py` searches existing PR comments for a matching header marker (`PRReviewHeader.REGULAR`), then edits the existing comment in-place via GitHub's API. Each subsequent `/review` replaces the previous review entirely. This is the sole deduplication mechanism — there is no semantic comparison of AI findings across runs. Each review is a completely independent LLM invocation.

The **token budgeting system** in `pr_processing.py` implements a priority-based compression pipeline. Files are sorted by language relevance (matching the PR's main language first), then patches are added sequentially until token limits are hit. Two thresholds control behavior: `OUTPUT_BUFFER_TOKENS_HARD_THRESHOLD` (absolute cutoff) and `OUTPUT_BUFFER_TOKENS_SOFT_THRESHOLD` (skip individual patches exceeding remaining budget). Large patches can be clipped or skipped entirely based on `large_patch_policy`. Context is extended with configurable surrounding lines (`patch_extra_lines_before=5`, `patch_extra_lines_after=1`) and optional dynamic context that includes enclosing function/class boundaries. Token budgeting does not change for incremental runs — the only difference is fewer files in the input set.

For automatic re-review on push, PR-Agent supports a `handle_push_trigger` configuration that runs specified commands (default: `/describe` and `/review`) on the `synchronize` webhook event. This runs a **full review**, not an incremental one. The model fallback strategy (`fallback_models`) tries alternative models if the primary one fails or hits token limits.

## GitHub Copilot and Claude Code Review take different but imperfect approaches

**GitHub Copilot Code Review** has no delta-aware mechanism whatsoever. The documentation explicitly states: "When re-reviewing a pull request, Copilot may repeat the same comments again, even if they have been dismissed with the 'Resolve conversation' button or downvoted with the 👎 button." Each review is an independent analysis. Re-review must be manually triggered via the Reviewers menu unless "Review new pushes" is enabled through repository rulesets. Developer reports indicate **non-deterministic results** across runs — one user documented getting 5, then 6, then 3 comments across three consecutive reviews of the same unchanged PR. The auto-review trigger reportedly fails roughly one-third of the time. Comments are posted incrementally as analysis completes rather than batched, causing confusion when the "finished reviewing and left 9 comments" notification precedes visible comments.

Recent improvements show Copilot moving toward better incremental handling. A March 2026 blog post describes capabilities to "maintain memory across reviews" through an agentic architecture overhaul, and comment clustering that groups multiple comments about the same pattern into single units. The system now uses multi-line comment ranges instead of single-line pins. These changes drove **8.1% improvement in positive feedback** and Copilot now accounts for over 1 in 5 code reviews on GitHub with 60+ million reviews conducted.

**Anthropic's Claude Code Review** (launched March 9, 2026) offers three configurable trigger modes: "Once after PR creation," "After every push," and "Manual" (triggered by `@claude review`). In the "After every push" mode, the system **auto-resolves comment threads when flagged issues are fixed** in subsequent commits — a meaningful improvement over Copilot's approach. However, the underlying analysis appears to be a full re-review each time. The docs state: "When a review runs, multiple agents analyze the diff and surrounding code in parallel on Anthropic infrastructure."

Claude Code Review's architecture runs **five independent specialized agents** in parallel, each examining code from a different angle (CLAUDE.md compliance, bug detection, git history context, previous PR comments, code comment verification). Results pass through a verification step and are deduplicated and ranked by severity on a **0-100 confidence scale** (default threshold: 80). Average review takes approximately 20 minutes and costs **$15–$25** — making "after every push" mode expensive for high-frequency workflows. A known issue (GitHub #4539) reports the reviewer sometimes "analyzes commits individually rather than the final cumulative state," flagging code from early commits already modified in later ones.

**Ellipsis** reviews every commit of every PR automatically. Their architecture uses multiple parallel "Comment Generators" (specialized LLM agents) with a **deduplication filter** that removes similar comments — but this operates within a single review pass across parallel generators, not documented as working across review passes. They use incremental vector indexing with SHA-based chunk IDs via Turbopuffer, re-embedding only changed chunks. **Greptile** is explicitly a "single-pass system — each review is independent, not iterative." When `triggerOnUpdates` is enabled, each new commit triggers a complete independent review with no cross-review state management.

## Academic incremental analysis offers a blueprint LLM tools haven't adopted

The gap between traditional incremental static analysis and LLM-based review is stark. Established systems maintain precise, deterministic state that enables true incrementality. LLM-based tools operate stochastically with no formal state representation, making the problem fundamentally harder.

**Facebook/Meta Infer** demonstrates the gold standard for differential analysis. Built on separation logic and bi-abduction, Infer computes **compositional procedure summaries** (pre/postcondition pairs) independently per function. These summaries are the key incrementality enabler: when a file changes, only changed procedures and their dependents need re-analysis. The recommended CI workflow uses `--reactive` mode (prevents deletion of previous results), `--changed-files-index` (specifies which files changed), and `reportdiff` to classify findings into three categories: `introduced.json` (new issues), `fixed.json` (resolved issues), and `preexisting.json` (unchanged issues). This **introduced/fixed/preexisting classification** is directly applicable to AI review but no production tool implements it. The impact of diff-time deployment was dramatic: the same analysis went from **~0% fix rate** in batch mode to **>70% fix rate** when deployed at diff time — a lesson Peter O'Hearn calls "continuous reasoning."

**SonarQube** implements two complementary mechanisms. The "skip unchanged files" mechanism supplies only modified files to analyzers for languages analyzable independently (CSS, Go, Ruby), while languages with cross-file dependencies (Java, JavaScript, C#) use optimized analysis with `SensorContext.canSkipUnchangedFiles()`. An analysis cache stores metadata server-side per-branch, with cache invalidation tracking cross-file dependencies, quality profile changes, and build settings. The **"New Code Period" concept** focuses quality enforcement exclusively on changed code — issues in old code are tracked but don't fail quality gates. This maps cleanly to focusing LLM review on changed regions only.

Several academic papers offer directly applicable techniques:

- **Reviser** (Arzt & Bodden, ICSE 2014) automatically extracts changed program parts and re-analyzes only those plus dependencies, achieving **up to 80% time savings** over full recomputation — applicable to any IDE/IFDS-based analysis.
- **Szabó's iQL** (ESEC/FSE 2023) explored incrementalizing GitHub's CodeQL, achieving **~15-second update times** for commits affecting up to 1,000 lines, though with steep initialization costs (~15 minutes, ~20 GB memory). CodeQL does not support incremental scans in production.
- **Conway et al.** (CAV 2005) demonstrated that function-level incremental inter-procedural analysis achieves **8.2x speedup**, with efficiency increasing as programs grow larger because "the relative size of individual functions decreases."
- **Google's Tricorder** processes over 50,000 code review changes daily with stateless, change-based analysis. Almost all analyzers are intra-procedural. The critical design lesson: results shown primarily on changed lines, with a "NOT USEFUL" button for continuous false-positive calibration. Google's failed FindBugs deployment versus successful Tricorder deployment proved that UX and false-positive management matter more than analysis sophistication.

The key insight for AI review design is that **compositional summaries** (from Infer), **report diffing** (introduced/fixed/preexisting classification), **change-period scoping** (from SonarQube), and **dependency-aware context selection** (from Reviser and Microsoft's CodePlan) are all proven techniques that no LLM-based tool fully implements. The fundamental challenge is that LLM analysis is non-deterministic — the same input may produce different output, making traditional cache-invalidation and deterministic summary reuse impossible.

## Comment lifecycle and notification fatigue are the critical UX failures

The engineering challenge of comment lifecycle management is compounded by **platform API constraints**. GitHub's PR review comment API anchors comments to positions within diff hunks using a `position` value counting lines from the hunk header. Comments can only be placed on lines within the diff — attempting otherwise returns a 422 error. This was documented as a real limitation by the PR-Agent project (Issue #592). Force-pushes are particularly destructive: they create entirely new commit SHAs, causing all comments on old commits to become "outdated" and collapsed. GitHub's long-standing Issue #443 ("Don't lose commit-related comments on rebase") remains unresolved.

Developer frustration with noise is quantified and severe. Multiple independent assessments converge on **60–80% noise rates** for AI code review comments. A DEV Community analysis states: "Most AI code review tools generate 10–20 comments per PR. The problem? 80% are noise." Independent testing found approximately 28% of CodeRabbit comments were noise or incorrect. One developer running four AI reviewers simultaneously documented receiving **72 comments on a single architectural PR** as the only human participant. The Stack Overflow 2025 Developer Survey (49,000+ respondents) found trust in AI accuracy dropped from 40% to **29%**, with 45% citing "AI solutions that are almost right, but not quite" as the top frustration.

A CodeRabbit user on AWS Marketplace captured the incremental review failure mode precisely: "It is unstoppable in its suggestions, providing comments and change requests even to the code it suggested in previous iterations, so the process can run forever." This infinite-loop dynamic — AI suggests change, developer implements it, AI suggests reverting or modifying the change — is the pathological case of stateless re-review.

Tools are converging on several UX solutions. **Gitar** pioneered a "living dashboard" — a single comment that updates in-place and auto-resolves inline comments when issues are fixed. **Sourcery** marks addressed suggestions with a "✅ Addressed" indicator. **CodeRabbit** allows chat-based feedback training through a "Learnings" system. GitHub Copilot now clusters comments and uses multi-line ranges. Most tools support path-based filtering via configuration files, severity thresholds, and 👍/👎 feedback mechanisms. The empirical evidence from a study of 22,000+ AI review comments across 178 repositories found that **hunk-level review tools** (focused on specific changed regions) produce more actionable comments than file-level or PR-level tools, and manually-triggered reviews have higher adoption than automatic ones.

## Conclusions for building an incremental review system

The research points to a clear architectural blueprint that no existing tool fully implements. An effective incremental AI review system should incorporate these specific mechanisms:

**Adopt Infer's report-diffing pattern.** Maintain a baseline of review findings per PR. On each re-review, classify every finding as introduced (new), fixed (resolved by new commits), or preexisting (unchanged). Only surface introduced findings to the developer. This single technique would eliminate most duplicate comment complaints. Store the baseline as a structured hidden comment on the PR (CodeRabbit's privacy-preserving approach) or in a lightweight server-side cache keyed by PR ID and commit SHA.

**Implement hybrid delta-plus-context review.** Review only changed files but include cached summaries of previously-reviewed files as LLM context. This captures cross-cutting issues (a new commit in file A that breaks an assumption in previously-reviewed file B) without the noise of full re-review. SonarQube's language-dependent strategy is instructive: files in languages with strong cross-file dependencies (Java, TypeScript) need broader context; files in languages analyzable independently (Go, Ruby) need less.

**Build comment lifecycle management against platform constraints.** Track each posted comment's commit SHA, file path, and line range. On re-review, compute a mapping between old and new diff positions. Auto-resolve comments on code that changed (the developer likely addressed the feedback). Mark comments on unchanged code as preexisting and suppress re-posting. Use GitHub's edit-comment API to update a single summary comment rather than posting new ones. Handle force-pushes by detecting SHA discontinuity and performing a full re-review with report diffing against the previous findings.

**Respect the evidence on signal-to-noise ratio.** The data consistently shows that fewer, higher-confidence comments outperform voluminous output. Set a high severity threshold by default (Claude Code Review's 80/100 confidence cutoff is a reasonable starting point). Implement a per-repository feedback loop where 👎 reactions increase the suppression threshold for similar comment patterns. Google Tricorder's lesson — that FindBugs failed while Tricorder succeeded due to UX, not analysis quality — should guide every design decision. Target **≤5 substantive comments per review** based on GitHub Copilot's production metrics showing this as the effective range.

**Use function-level granularity as the change unit.** Conway et al.'s finding that function-level incremental analysis achieves 8.2x speedup with increasing efficiency at scale translates directly: track review status per function, not per file. A file may change in ways that don't affect a previously-reviewed function. Use tree-sitter or similar AST parsing to detect which functions actually changed semantically (not just formatting), and only re-review those functions plus their direct callers/callees.

The honest assessment: no production tool today fully solves incremental AI review. CodeRabbit comes closest with genuine incremental file detection and persistent state, but still lacks automatic comment resolution and report diffing. The path forward requires combining proven incremental static analysis techniques (compositional summaries, differential workflows, change-period scoping) with LLM-specific innovations (confidence calibration, feedback loops, non-determinism-aware caching). The tools that solve this well will win — the 29% trust figure from Stack Overflow shows the current approach is not sustainable.
